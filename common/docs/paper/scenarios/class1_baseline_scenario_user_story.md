# Class 1 Baseline Scenario User Story
## 사용자가 단순 입력으로 조명 제어 도움을 요청하는 상황

## 1. 이 문서가 설명하는 JSON 파일

이 문서는 다음 scenario JSON 파일을 실제 사용환경에서 일어날 수 있는 상황으로 풀어 설명한다.

```text
integration/scenarios/class1_baseline_scenario_skeleton.json
```

이 JSON 파일은 사용자가 집 안에서 **낮은 위험도의 생활 보조 요청**을 했을 때, 시스템이 이를 긴급상황이 아닌 일반 보조 흐름으로 인식하고 처리하는지를 검토하기 위한 시나리오다.

여기서는 사용자가 침대나 휠체어에 있는 상태에서 버튼을 한 번 눌러 **조명을 켜거나 끄는 도움을 요청하는 상황**을 예로 든다.

이 문서는 이후 실제 노드를 개발할 때 다음을 확인하는 기준으로도 사용할 수 있다.

```text
- 어떤 노드가 사용자의 단순 입력을 감지해야 하는가
- 조명 상태와 주변 환경 정보는 어떤 노드가 제공해야 하는가
- 시스템은 긴급상황과 일반 생활 보조 요청을 어떻게 구분하는가
- 사용자가 기대하는 조명 제어 결과는 무엇인가
```

---

## 2. 발생 상황

사용자는 집 안에서 혼자 있거나, 보호자가 바로 옆에 없는 상황이라고 가정한다.

사용자는 침대에 누워 있거나 휠체어에 앉아 있다. 몸을 자유롭게 움직이기 어렵기 때문에 벽 스위치까지 이동하거나, 스마트폰을 들고 앱을 열어 조명을 조작하는 일이 쉽지 않다.

방 안은 어둡거나, 반대로 사용자가 휴식을 취하기 위해 조명을 끄고 싶은 상황일 수 있다.

사용자 가까이에는 손이나 팔의 작은 움직임으로 누를 수 있는 버튼 입력 장치가 있다. 사용자는 복잡한 음성 명령이나 스마트폰 조작 대신 버튼을 한 번 눌러 조명 제어 도움을 요청한다.

이 시나리오의 핵심 상황은 다음과 같다.

```text
사용자는 침대나 휠체어에 있고,
스마트폰 조작이나 벽 스위치 조작이 어렵고,
버튼 한 번으로 조명 제어 도움을 요청한다.
```

---

## 3. 시스템이 인식해야 하는 상황

시스템은 사용자의 버튼 입력을 단독으로만 보지 않고, 현재 집 안의 상태와 함께 확인해야 한다.

예를 들어 시스템은 다음과 같은 정보를 함께 확인한다.

```text
- 사용자가 버튼을 한 번 눌렀다.
- 현재 사용자가 있는 공간의 조명 상태를 확인할 수 있다.
- 방 안의 밝기나 사용자의 위치 정보를 참고할 수 있다.
- 화재, 가스, 낙상 같은 긴급 상황은 감지되지 않았다.
- 방문자 호출이나 현관 대응 상황도 아니다.
```

이 정보를 종합하면, 시스템은 사용자의 버튼 입력을 **일상적인 조명 제어 도움 요청**으로 해석할 수 있다.

여기서 중요한 점은 조명 상태나 환경 정보가 시스템 안에 저절로 존재하는 것이 아니라는 점이다. 조명 상태는 Lighting Control Node 또는 Device-State Reporting Function가 제공해야 하고, 방 안의 밝기나 점유 여부는 Context Node가 제공해야 한다.

Mac mini Edge Hub는 여러 노드가 제공한 정보를 종합해 사용자의 요청을 해석한다.

---

## 4. 정보 출처 노드 및 상태 정보

이 시나리오에서 시스템은 사용자의 버튼 입력과 주변 상태 정보를 함께 사용한다. 각 정보의 출처를 명확히 해야 실제 노드 개발 시 어떤 기능이 필요한지 알 수 있다.

| 항목 | 정보를 제공하는 노드/구성요소 | 제공되는 정보 | 시나리오 내 상태 |
|---|---|---|---|
| 사용자 입력 | Bounded Input Node | 사용자가 버튼을 눌렀다는 입력 이벤트 | 단일 버튼 입력 발생 |
| 입력 의미 | Bounded Input Node 또는 Input Context Mapper | 단일 입력이 일반 생활 보조 요청이라는 정보 | 조명 제어 요청 후보 |
| 환경 정보 | Context Node | 조도, 점유 여부, 사용자가 있는 공간 등 주변 환경 정보 | 사용자가 실내에 있고 조명 제어가 필요한 상황 |
| 조명 상태 | Lighting Control Node 또는 Device-State Reporting Function | 조명이 현재 켜져 있는지 꺼져 있는지에 대한 상태 정보 | 현재 조명 상태 확인 가능 |
| 긴급상황 여부 | Emergency Node | 화재, 가스, 낙상 등 위험 상황 감지 정보 | 위험 상황 없음 |
| 방문자 여부 | Doorbell/Visitor Context Node | 현관 호출 또는 방문자 감지 정보 | 방문자 상황 아님 |
| 요청 해석 | Mac mini Edge Hub | 입력과 상태를 종합한 요청 해석 결과 | 일반 생활 보조 요청으로 해석 |
| 조명 실행 | Lighting Control Node | 조명 켜기 또는 끄기 동작 수행 및 실행 결과 | 조명 제어 수행 |
| 사용자 피드백 | TTS/Voice Output | 처리 결과를 사용자에게 알려주는 음성 안내 | 필요 시 조명 처리 결과 안내 |
| 처리 기록 | Audit Log | 입력, 해석, 실행 결과 기록 | 처리 결과 기록 |

이 표에서 조명 상태와 환경 정보는 단순한 참고 문구가 아니라 실제 노드가 제공해야 하는 데이터다.

```text
조명 상태
→ Lighting Control Node 또는 Device-State Reporting Function가 제공하는 기기 상태 정보

환경 정보
→ Context Node가 제공하는 조도, 점유 여부, 공간 상태 정보
```

따라서 실제 개발에서는 “사용자가 버튼을 눌렀다”뿐 아니라, **그 버튼 입력을 해석하는 데 필요한 주변 상태를 어떤 노드가 제공할 것인가**까지 함께 설계해야 한다.

## 5. 노드별 개발 관점 정리

이 시나리오를 구현하기 위해 필요한 노드와 구성요소는 다음과 같다.

| 노드/구성요소 | 개발 시 필요한 기능 | 이 시나리오에서의 역할 |
|---|---|---|
| Bounded Input Node | 버튼 입력 감지, debounce 처리, 입력 이벤트 생성 | 사용자가 버튼을 한 번 눌렀다는 사실을 시스템에 전달 |
| Input Context Mapper | 입력 패턴과 현재 상황을 연결해 가능한 요청 후보를 구성 | 단일 입력을 조명 제어 요청 후보로 해석하는 데 도움 |
| Context Node | 조도, 점유 여부, 사용자 위치 또는 공간 정보 제공 | 조명 제어가 자연스러운 상황인지 판단하는 데 도움 |
| Lighting Control Node | 조명 켜기/끄기 수행, 현재 조명 상태 보고, 실행 결과 ACK 반환 | 조명 상태를 보고하고 요청된 조명 동작을 수행 |
| Device-State Reporting Function | 조명 등 기기 상태를 주기적 또는 이벤트 기반으로 보고 | Lighting Control Node가 상태 보고를 직접 하지 않는 경우 필요 |
| Emergency Node | 화재, 가스, 낙상 등 위험 상황 감지 | 이 시나리오에서는 위험 상황이 없음을 나타냄 |
| Doorbell/Visitor Context Node | 현관 호출 또는 방문자 감지 여부 제공 | 이 시나리오에서는 방문자 상황이 아님을 나타냄 |
| Mac mini Edge Hub | 입력 수신, 상태 종합, 요청 해석, 실행 판단, 로그 기록 | 버튼 입력과 상태 정보를 바탕으로 조명 제어 요청을 처리 |
| TTS/Voice Output | 사용자에게 처리 결과를 음성으로 안내 | 필요 시 “조명을 켰습니다.” 또는 “조명을 껐습니다.” 안내 |
| Audit Log | 입력, 해석, 실행 결과 저장 | 시스템 처리 결과를 나중에 확인할 수 있도록 기록 |

## 6. 처리 과정

### 6.1 사용자가 버튼을 한 번 누른다

사용자는 침대나 휠체어에 있는 상태에서 버튼을 한 번 누른다.

Bounded Input Node는 이 입력을 감지한다.

이때 Bounded Input Node는 단순한 전기 신호를 시스템이 이해할 수 있는 입력 이벤트로 변환한다.

예를 들어 다음과 같은 의미의 이벤트가 생성된다.

```text
사용자 버튼 입력 1회 발생
```

---

### 6.2 시스템이 현재 환경 정보를 확인한다

Context Node는 현재 방 안의 상태를 제공한다.

예를 들어 시스템은 다음과 같은 정보를 참고할 수 있다.

```text
사용자가 실내에 있다.
방 안의 조도가 낮다.
현재 공간에서 조명 제어가 필요한 상황일 수 있다.
```

이 정보는 사용자의 단일 버튼 입력을 조명 제어 요청으로 해석하는 데 도움을 준다.

---

### 6.3 조명 상태가 확인된다

Lighting Control Node 또는 Device-State Reporting Function는 현재 조명 상태를 제공한다.

예를 들어 조명이 꺼져 있다면 시스템은 사용자가 조명을 켜고 싶어 한다고 판단할 수 있다.

반대로 조명이 켜져 있고 사용자가 휴식 중인 상황이라면 조명을 끄고 싶어 하는 요청일 수도 있다.

```text
현재 조명 상태 확인
→ 켜짐 또는 꺼짐
```

이처럼 같은 버튼 입력이라도 현재 조명 상태와 환경 정보에 따라 해석이 달라질 수 있다.

---

### 6.4 긴급상황과 방문자 상황이 아닌지 확인한다

Emergency Node는 화재, 가스, 낙상 등 위험 상황이 있는지 확인한다.

Doorbell/Visitor Context Node는 방문자 호출이나 현관 대응 상황이 있는지 확인한다.

이 시나리오에서는 다음과 같은 상태를 가정한다.

```text
긴급상황 감지 없음
방문자 상황 아님
```

따라서 시스템은 이 입력을 긴급 대응이나 방문자 대응이 아니라, 일반적인 생활 보조 요청으로 처리할 수 있다.

---

### 6.5 Mac mini Edge Hub가 정보를 종합한다

Mac mini Edge Hub는 다음 정보를 함께 확인한다.

```text
사용자가 버튼을 한 번 눌렀다.
사용자가 실내에 있다.
현재 조명 상태를 확인할 수 있다.
긴급상황은 없다.
방문자 상황도 아니다.
```

이 정보를 바탕으로 시스템은 사용자의 요청을 다음과 같이 해석할 수 있다.

```text
사용자는 조명 제어 도움을 요청하고 있다.
```

---

### 6.6 조명 제어 동작이 실행된다

Mac mini Edge Hub가 조명 제어 요청으로 판단하면, Lighting Control Node에 조명 제어 동작을 요청한다.

조명이 꺼져 있고 사용자가 조명을 켜고 싶어 하는 상황이면 조명을 켠다.

조명이 켜져 있고 사용자가 조명을 끄고 싶어 하는 상황이면 조명을 끈다.

Lighting Control Node는 조명 동작을 수행한 뒤 실행 결과를 반환한다.

```text
조명 제어 수행
조명 상태 변경 결과 확인
```

---

### 6.7 사용자에게 처리 결과를 알려준다

필요한 경우 TTS/Voice Output이 사용자에게 처리 결과를 안내한다.

예를 들어 다음과 같은 음성 안내가 가능하다.

```text
“조명을 켰습니다.”
```

또는 상황에 따라 다음과 같은 안내도 가능하다.

```text
“조명을 껐습니다.”
```

이 안내는 사용자가 화면을 보지 않아도 시스템이 동작했는지 알 수 있게 해준다.

---

### 6.8 처리 결과를 기록한다

Audit Log는 이 과정에서 발생한 주요 정보를 기록한다.

예를 들어 다음과 같은 내용이 기록될 수 있다.

```text
- 버튼 입력이 발생한 시간
- 입력을 제공한 노드
- 참조된 환경 정보
- 참조된 조명 상태
- 시스템이 조명 제어 요청으로 해석한 내용
- 조명 제어 실행 결과
- 사용자에게 안내가 제공되었는지 여부
```

이 기록은 나중에 시스템이 정상적으로 동작했는지 확인하거나, 실험 결과를 정리할 때 사용할 수 있다.

---

## 7. 예상되는 결과

이 시나리오에서 기대되는 결과는 다음과 같다.

사용자가 버튼을 한 번 누르면 시스템이 현재 상황을 확인하고, 조명 제어 요청으로 판단한 뒤 조명을 제어한다.

사용자 입장에서는 다음과 같은 흐름이 된다.

```text
사용자가 버튼을 한 번 누른다.
→ Bounded Input Node가 입력을 감지한다.
→ 시스템이 환경 정보와 조명 상태를 확인한다.
→ 시스템이 긴급상황이나 방문자 상황이 아님을 확인한다.
→ 시스템이 조명 제어 요청으로 해석한다.
→ 조명이 켜지거나 꺼진다.
→ 필요하면 처리 결과를 음성으로 안내한다.
→ 처리 과정이 기록된다.
```

## 8. 사용자에게 주는 의미

이 시나리오는 사용자가 복잡한 조작 없이 일상적인 스마트홈 기능을 사용할 수 있게 해준다.

지체장애인, 고령자, 또는 운동 기능이 제한된 사용자는 벽 스위치까지 이동하거나 스마트폰을 조작하는 일이 어려울 수 있다. 또한 음성 명령을 또렷하게 말하기 어려운 경우도 있다.

이때 작은 버튼 입력 하나만으로 조명 제어가 가능하다면, 사용자는 더 안전하고 독립적으로 생활할 수 있다.

이러한 방식은 특히 다음과 같은 사용자에게 도움이 될 수 있다.

```text
- 침대에서 일어나기 어려운 사용자
- 휠체어에서 벽 스위치까지 이동하기 어려운 사용자
- 손가락으로 스마트폰을 조작하기 어려운 사용자
- 음성 명령을 또렷하게 말하기 어려운 사용자
- 야간 이동 전 조명을 켜야 하는 고령자
```

## 9. 개발 관점에서 확인해야 할 점

이 문서는 실제 상황 설명을 중심으로 하지만, 노드 개발을 위해서는 다음 사항을 확인해야 한다.

```text
- Bounded Input Node는 버튼 입력을 안정적으로 감지할 수 있는가?
- Bounded Input Node는 단일 입력과 반복 입력을 구분할 수 있는가?
- Context Node는 조도, 점유 여부, 사용자가 있는 공간 정보를 제공할 수 있는가?
- Lighting Control Node는 현재 조명 상태를 보고할 수 있는가?
- Lighting Control Node는 조명 켜기/끄기 명령을 수행하고 결과를 반환할 수 있는가?
- Device-State Reporting Function는 조명 상태를 최신 상태로 유지할 수 있는가?
- Emergency Node는 긴급상황이 없다는 상태도 구분할 수 있는가?
- Doorbell/Visitor Context Node는 방문자 상황이 아님을 제공할 수 있는가?
- Mac mini Edge Hub는 여러 노드의 정보를 종합해 요청을 해석할 수 있는가?
- TTS/Voice Output은 처리 결과를 사용자에게 안내할 수 있는가?
- Audit Log는 입력, 해석, 실행 결과를 기록할 수 있는가?
```

이 체크리스트는 Class 1 조명 보조 시나리오를 실제 시스템으로 구현할 때 필요한 최소 개발 항목을 정리한 것이다.

## 10. 이 시나리오의 핵심

이 시나리오의 핵심은 다음과 같다.

```text
사용자가 작은 움직임으로 버튼을 한 번 누르면,
여러 노드가 현재 상황 정보를 제공하고,
Mac mini Edge Hub가 그 정보를 종합해,
사용자가 필요로 하는 조명 제어를 수행한다.
```

즉, 이 시나리오는 스마트홈 시스템이 사용자의 제한된 입력을 받아, 긴급상황이 아닌 일상적인 생활 보조 요청을 안전하게 처리하는 기본 Class 1 시나리오다.
