# Class 0 E002 Scenario User Story
## 사용자가 버튼을 세 번 눌러 긴급 도움을 요청하는 상황

## 1. 이 문서가 설명하는 JSON 파일

이 문서는 다음 scenario JSON 파일을 실제 사용환경에서 일어날 수 있는 상황으로 풀어 설명한다.

```text
integration/scenarios/class0_e002_scenario_skeleton.json
```

이 JSON 파일은 사용자가 위급한 상황에서 **bounded input 장치를 짧은 시간 안에 세 번 누르는 방식으로 긴급 도움을 요청했을 때**, 시스템이 이를 긴급상황으로 인식하고 즉시 대응 흐름으로 전환하는지를 검토하기 위한 시나리오다.

여기서는 사용자가 넘어질 위험을 느끼거나 갑작스러운 신체 이상을 느꼈지만, 스마트폰 조작이나 음성 명령이 어려워 버튼을 세 번 눌러 긴급 도움을 요청하는 상황을 예로 든다.

이 문서는 이후 실제 노드를 개발할 때 다음을 확인하는 기준으로도 사용할 수 있다.

```text
- 버튼 입력 노드는 연속 입력 패턴을 안정적으로 감지할 수 있는가
- 세 번 누름 입력을 일반 조명 요청과 긴급 요청으로 구분할 수 있는가
- 긴급 입력 정보는 어떤 방식으로 중앙 시스템에 전달되는가
- 사용자가 말하거나 이동하기 어려운 상황에서 어떤 결과가 기대되는가
```

---

## 2. 발생 상황

사용자는 집 안에 혼자 있거나 보호자가 바로 옆에 없는 상황이라고 가정한다.

사용자는 침대에서 휠체어로 이동하려고 하거나, 욕실이나 거실에서 자세를 바꾸는 중 갑자기 몸의 균형을 잃을 것 같은 느낌을 받는다. 또는 호흡이 불편하거나 갑자기 어지러움을 느껴 즉시 도움을 요청해야 할 수도 있다.

하지만 사용자는 스마트폰을 꺼내 앱을 열기 어렵고, 음성 명령을 또렷하게 말하기도 어렵다. 몸을 크게 움직이는 것도 부담스럽다.

사용자 가까이에는 작은 움직임으로 누를 수 있는 버튼 입력 장치가 있다. 사용자는 긴급 도움을 요청하기 위해 버튼을 빠르게 세 번 누른다.

이 시나리오의 핵심 상황은 다음과 같다.

```text
사용자는 즉시 도움이 필요한 상황에 놓여 있고,
말하거나 이동하기 어려우며,
버튼을 세 번 누르는 방식으로 긴급 도움을 요청한다.
```

---

## 3. 시스템이 인식해야 하는 상황

시스템은 버튼 입력이 한 번 들어왔다고 해서 항상 같은 의미로 해석하면 안 된다.

일반적인 한 번 입력은 조명 켜기와 같은 생활 보조 요청일 수 있지만, 짧은 시간 안에 세 번 반복된 입력은 사용자의 긴급 신호일 수 있다.

예를 들어 시스템은 다음과 같은 정보를 함께 확인한다.

```text
- 사용자가 버튼을 빠르게 세 번 눌렀다.
- 이 입력은 일반적인 단일 버튼 입력과 다르다.
- 사용자가 직접 음성으로 도움을 요청하지 못할 수 있다.
- 사용자는 즉각적인 도움이나 확인이 필요한 상태일 수 있다.
- 현재 입력은 일반적인 조명 제어 요청이 아니라 긴급 도움 요청으로 처리되어야 한다.
```

이 정보를 종합하면, 시스템은 이 상황을 **사용자의 명시적 긴급 도움 요청**으로 처리해야 한다.

여기서 중요한 점은 triple-hit 정보가 단순한 버튼 신호 세 개의 나열이 아니라는 점이다. Bounded Input Node는 세 번의 입력이 일정한 시간 안에 발생했는지 감지하고, 이를 하나의 긴급 입력 이벤트로 구성해야 한다.

Mac mini Edge Hub는 이 긴급 입력 이벤트를 수신한 뒤, 일반적인 생활 보조 요청보다 긴급 대응 흐름을 우선해야 한다.

## 4. 정보 출처 노드 및 상태 정보

이 시나리오에서 시스템은 사용자의 버튼 입력 패턴을 중심으로 긴급상황 여부를 판단한다. 동시에 사용자 주변 환경, 방문자 여부, 안내 출력, 기록 기능도 함께 고려된다.

| 항목 | 정보를 제공하는 노드/구성요소 | 제공되는 정보 | 시나리오 내 상태 |
|---|---|---|---|
| 긴급 버튼 입력 | Bounded Input Node | 사용자가 버튼을 짧은 시간 안에 세 번 눌렀다는 입력 패턴 | triple-hit 입력 발생 |
| 입력 패턴 판단 | Bounded Input Node 또는 Input Pattern Detector | 단일 입력인지, 반복 입력인지, 긴급 입력 패턴인지에 대한 판단 정보 | 긴급 도움 요청 패턴으로 판단 |
| 사용자 상태 | Bounded Input Node 또는 사용자 상태 정보 | 사용자가 직접 음성 명령이나 스마트폰 조작을 하기 어려운 상태와 관련된 정보 | 제한적 조작 가능 상태 |
| 환경 정보 | Context Node | 사용자가 있는 공간, 주변 환경, 점유 여부 등 | 사용자가 실내에 있음 |
| 긴급상황 여부 | Emergency Node 또는 Mac mini Edge Hub | triple-hit 입력을 긴급 이벤트로 처리해야 하는지에 대한 정보 | 긴급 이벤트 발생 |
| 방문자 여부 | Doorbell/Visitor Context Node | 현관 호출 또는 방문자 감지 정보 | 방문자 상황 아님 |
| 요청/상황 해석 | Mac mini Edge Hub | 버튼 세 번 입력을 종합한 처리 판단 | 긴급 도움 요청으로 해석 |
| 사용자 경고/확인 | TTS/Voice Output 또는 Warning Output Node | 사용자에게 긴급 요청이 접수되었음을 알리는 안내 | 접수 안내 가능 |
| 보호자/외부 알림 | Caregiver Notification 또는 Escalation Interface | 보호자나 관리자에게 긴급 도움 요청 전달 | 필요 시 긴급 알림 가능 |
| 처리 기록 | Audit Log | 입력 패턴, 처리 판단, 알림 결과 기록 | 긴급 요청 처리 결과 기록 |

이 표에서 triple-hit 입력은 단순한 반복 입력이 아니라, 실제 사용자가 위급함을 표현하기 위해 사용할 수 있는 긴급 신호다.

```text
Triple-hit 입력
→ Bounded Input Node 또는 Input Pattern Detector가 감지하는 긴급 입력 패턴

긴급 도움 요청 판단
→ Mac mini Edge Hub가 입력 패턴과 현재 상황을 바탕으로 수행하는 처리 결과
```

따라서 실제 개발에서는 “버튼 입력이 있다”만 확인하는 것이 아니라, **버튼 입력의 횟수, 시간 간격, 반복 패턴을 어떤 노드가 감지하고 어떻게 긴급 이벤트로 전달할 것인가**까지 설계해야 한다.

## 5. 노드별 개발 관점 정리

이 시나리오를 구현하기 위해 필요한 노드와 구성요소는 다음과 같다.

| 노드/구성요소 | 개발 시 필요한 기능 | 이 시나리오에서의 역할 |
|---|---|---|
| Bounded Input Node | 버튼 입력 감지, debounce 처리, 짧은 시간 내 반복 입력 감지 | 사용자가 버튼을 세 번 눌렀다는 사실을 감지 |
| Input Pattern Detector | 입력 횟수와 시간 간격 분석, triple-hit 패턴 판정 | 세 번 누름을 긴급 요청 패턴으로 구분 |
| Context Node | 사용자 위치, 점유 여부, 주변 환경 정보 제공 | 사용자가 실내에 있는 상황을 이해하는 데 도움 |
| Emergency Node | 긴급 입력 이벤트 생성 또는 전달 | triple-hit 입력을 긴급상황 이벤트로 전달 |
| Doorbell/Visitor Context Node | 현관 호출 또는 방문자 감지 여부 제공 | 이 시나리오에서는 방문자 상황이 아님을 나타냄 |
| Mac mini Edge Hub | 긴급 입력 이벤트 수신, 상황 판단, 긴급 대응 흐름 실행 | triple-hit 입력을 Class 0 긴급 도움 요청으로 처리 |
| TTS/Voice Output | 사용자에게 요청 접수 또는 위험 상황을 음성으로 안내 | 필요 시 “긴급 요청이 접수되었습니다.” 안내 |
| Warning Output Node | 경고음, 표시등, 화면 알림 등 보조 경고 출력 | 사용자가 요청이 전달되었음을 확인하도록 도움 |
| Caregiver Notification 또는 Escalation Interface | 보호자나 관리자에게 알림 전송 | 사용자의 긴급 도움 요청을 외부에 전달 |
| Audit Log | 입력 패턴, 판단, 알림, 처리 결과 기록 | 긴급 요청 처리 과정 기록 |

## 6. 처리 과정

### 6.1 사용자가 버튼을 빠르게 세 번 누른다

사용자는 위급함을 느끼지만 음성 명령이나 스마트폰 조작이 어렵다.

사용자는 가까이에 있는 버튼을 빠르게 세 번 누른다.

Bounded Input Node는 이 입력을 감지한다.

단순한 입력 이벤트가 아니라, 다음과 같은 의미의 입력 패턴이 생성된다.

```text
사용자 버튼 입력 3회 연속 발생
```

---

### 6.2 입력 패턴이 긴급 요청으로 판정된다

Bounded Input Node 또는 Input Pattern Detector는 세 번의 버튼 입력이 일정한 시간 안에 발생했는지 확인한다.

이 시나리오에서는 세 번의 입력이 긴급 도움 요청 패턴으로 판정된다.

```text
입력 패턴: triple-hit
판정 결과: 긴급 도움 요청
```

이 단계가 중요하다. 한 번 누름과 세 번 누름은 사용자 입장에서 완전히 다른 의미를 가질 수 있기 때문이다.

---

### 6.3 긴급상황 이벤트가 생성된다

Emergency Node 또는 입력 노드는 triple-hit 입력을 긴급상황 이벤트로 구성한다.

이 이벤트는 일반적인 조명 제어 요청이 아니라, 즉각적인 확인과 도움 연결이 필요한 신호로 취급된다.

이 시나리오에서 긴급상황 이벤트는 다음 입력 경로로 전달된다.

```text
safe_deferral/emergency/event
```

---

### 6.4 Mac mini Edge Hub가 긴급 이벤트를 수신한다

Mac mini Edge Hub는 triple-hit 긴급 이벤트를 수신한다.

이때 시스템은 이 입력을 일반적인 버튼 입력이나 조명 제어 요청으로 처리하지 않는다.

즉, 사용자가 버튼을 눌렀다는 사실보다 중요한 것은 **짧은 시간 안에 세 번 눌렀다는 패턴**이다.

---

### 6.5 시스템이 긴급 도움 요청으로 판단한다

Mac mini Edge Hub는 triple-hit 입력을 긴급 도움 요청으로 판단한다.

이때 시스템은 다음과 같은 점을 함께 고려할 수 있다.

```text
사용자가 반복 입력으로 도움을 요청했다.
사용자는 말하거나 스마트폰을 조작하기 어려울 수 있다.
긴급 확인 또는 보호자 알림이 필요할 수 있다.
일반적인 생활 보조 요청으로 처리하면 안 된다.
```

이 시나리오에서는 이 상황을 사용자가 직접 보낸 긴급 도움 요청으로 해석한다.

---

### 6.6 사용자에게 요청 접수 사실을 알려준다

필요한 경우 TTS/Voice Output 또는 Warning Output Node가 사용자에게 긴급 요청이 접수되었음을 알려준다.

예를 들어 다음과 같은 안내가 가능하다.

```text
“긴급 요청이 접수되었습니다.”
```

또는 버튼 장치의 LED, 경고음, 화면 표시 등으로 사용자가 요청이 전달되었음을 확인할 수 있다.

이 안내는 매우 중요하다. 사용자는 자신의 요청이 실제로 전달되었는지 알기 어려울 수 있기 때문이다.

---

### 6.7 보호자 또는 관리자에게 알림을 보낼 수 있다

시스템은 보호자나 관리자에게 긴급 도움 요청을 전달할 수 있다.

예를 들어 다음과 같은 정보가 전달될 수 있다.

```text
- 긴급 버튼 입력이 발생한 시간
- 입력이 발생한 위치 또는 연결된 노드
- 사용자가 triple-hit 입력을 보냈다는 사실
- 사용자가 도움이 필요할 수 있다는 상태
- 시스템이 긴급상황으로 처리했다는 사실
```

이 단계는 사용자가 직접 전화를 걸거나 메시지를 보내기 어려운 상황에서 외부 도움을 빠르게 연결하기 위한 것이다.

---

### 6.8 처리 결과를 기록한다

Audit Log는 이 과정에서 발생한 주요 정보를 기록한다.

예를 들어 다음과 같은 내용이 기록될 수 있다.

```text
- 버튼 입력이 발생한 시간
- 입력을 제공한 노드
- 입력 횟수와 입력 패턴
- 시스템이 긴급 도움 요청으로 판단한 내용
- 사용자에게 접수 안내가 제공되었는지 여부
- 보호자 또는 관리자에게 알림이 전송되었는지 여부
- 이후 처리 결과
```

이 기록은 나중에 시스템이 정상적으로 긴급 요청을 처리했는지 확인하거나, 실험 결과를 분석할 때 사용할 수 있다.

## 7. 예상되는 결과

이 시나리오에서 기대되는 결과는 다음과 같다.

사용자가 버튼을 빠르게 세 번 누르면 시스템은 이를 일반적인 생활 보조 요청으로 처리하지 않고, 긴급 도움 요청으로 인식한다.

사용자 입장에서는 다음과 같은 흐름이 된다.

```text
사용자가 버튼을 빠르게 세 번 누른다.
→ Bounded Input Node가 입력 패턴을 감지한다.
→ 시스템이 triple-hit 입력을 긴급 도움 요청으로 판정한다.
→ Mac mini Edge Hub가 긴급 이벤트를 수신한다.
→ 사용자에게 긴급 요청 접수 사실을 알려준다.
→ 필요하면 보호자나 관리자에게 알림이 전달된다.
→ 처리 과정이 기록된다.
```

## 8. 사용자에게 주는 의미

이 시나리오는 사용자가 말하거나 움직이기 어려운 상태에서도 긴급 도움을 요청할 수 있게 해준다.

지체장애인, 고령자, 또는 운동 기능이 제한된 사용자는 위급한 상황에서 스마트폰을 조작하거나 큰 소리로 도움을 요청하기 어려울 수 있다.

이때 사용자가 할 수 있는 작은 움직임, 예를 들어 버튼을 세 번 누르는 동작만으로도 시스템이 긴급상황을 인식하고 보호자에게 연결할 수 있다.

이러한 방식은 특히 다음과 같은 사용자에게 도움이 될 수 있다.

```text
- 갑작스러운 어지러움이나 호흡 불편으로 말하기 어려운 사용자
- 침대나 휠체어에서 빠르게 이동하기 어려운 사용자
- 스마트폰을 잡고 조작하기 어려운 사용자
- 낙상 위험을 느꼈지만 아직 센서가 감지하지 못한 상황의 사용자
- 혼자 있을 때 즉시 보호자에게 도움을 요청해야 하는 사용자
```

---

## 9. 개발 관점에서 확인해야 할 점

이 문서는 실제 상황 설명을 중심으로 하지만, 노드 개발을 위해서는 다음 사항을 확인해야 한다.

```text
- Bounded Input Node는 버튼 입력을 안정적으로 감지할 수 있는가?
- Bounded Input Node는 debounce 처리를 통해 잘못된 중복 입력을 줄일 수 있는가?
- Input Pattern Detector는 짧은 시간 안의 세 번 입력을 정확히 구분할 수 있는가?
- 단일 입력과 triple-hit 입력을 서로 다른 의미로 처리할 수 있는가?
- Emergency Node는 triple-hit 입력을 긴급 이벤트로 전달할 수 있는가?
- Mac mini Edge Hub는 triple-hit 이벤트를 일반 입력보다 우선 처리할 수 있는가?
- TTS/Voice Output은 긴급 요청 접수 사실을 사용자에게 안내할 수 있는가?
- Warning Output Node는 LED, 소리, 화면 등으로 접수 상태를 알려줄 수 있는가?
- Caregiver Notification 또는 Escalation Interface는 보호자에게 필요한 정보를 전달할 수 있는가?
- Audit Log는 입력 횟수, 입력 패턴, 판단, 알림 결과를 기록할 수 있는가?
```

이 체크리스트는 triple-hit 긴급 입력을 실제 시스템으로 구현할 때 필요한 최소 개발 항목을 정리한 것이다.

## 10. 이 시나리오의 핵심

이 시나리오의 핵심은 다음과 같다.

```text
사용자가 말하거나 이동하기 어려운 상황에서
버튼을 세 번 누르면,
시스템이 이를 긴급 도움 요청으로 인식하고,
사용자와 보호자에게 필요한 대응을 연결한다.
```

즉, 이 시나리오는 스마트홈 시스템이 사용자의 제한된 움직임을 긴급 신호로 해석해, 사용자가 직접 말하거나 스마트폰을 조작하지 않아도 도움을 요청할 수 있게 하는 기본 긴급 입력 시나리오다.
