# Class 2 Insufficient Context Scenario User Story
## 정보가 부족한 사용자 입력에서 후보 제시 후 Class 0 또는 Class 1로 전이되는 상황

## 1. 이 문서가 설명하는 JSON 파일

이 문서는 다음 scenario JSON 파일을 실제 사용환경에서 일어날 수 있는 상황으로 풀어 설명한다.

```text
integration/scenarios/class2_insufficient_context_scenario_skeleton.json
```

이 JSON 파일은 사용자가 집 안에서 도움을 요청했지만, 현재 정보만으로는 그 요청을 **긴급상황인지, 낮은 위험도의 생활 보조 요청인지, 또는 단순 확인이 필요한 상황인지** 바로 확정하기 어려운 경우를 다룬다.

이 시나리오에서 Class 2는 최종 실패 상태가 아니다. Class 2는 정보가 부족한 상태에서 사용자의 의도를 더 안전하게 확인하기 위한 **clarification / transition state**이다.

즉, 시스템은 사용자의 입력을 바로 실행하거나 무시하지 않고, 가능한 후보를 제시한 뒤 사용자 또는 보호자의 확인을 받아 다음 상태로 전이한다.

```text
Class 2
→ 후보 제시 및 확인
→ Class 1 생활 보조 요청으로 전이
또는
→ Class 0 긴급상황으로 전이
또는
→ 계속 불명확하면 안전 보류 또는 보호자 확인
```

이 문서는 이후 실제 노드를 개발할 때 다음을 확인하는 기준으로도 사용할 수 있다.

```text
- 어떤 상황에서 사용자의 의도를 바로 확정할 수 없는가
- LLM은 어떤 방식으로 가능한 도움 후보를 제시해야 하는가
- 사용자는 후보 중 하나를 어떻게 선택하거나 확인할 수 있는가
- 어떤 응답이 Class 1 생활 보조 요청으로 이어지는가
- 어떤 응답이 Class 0 긴급상황으로 이어지는가
- 응답이 없거나 계속 불명확하면 어떻게 안전하게 보류할 것인가
```

## 2. 발생 상황

사용자는 침대에 누워 있거나 휠체어에 앉아 있다.

사용자는 어떤 도움이 필요해서 버튼을 한 번 누른다. 하지만 현재 시스템은 사용자가 정확히 무엇을 원하는지 확실히 알 수 없다.

예를 들어 다음과 같은 상황일 수 있다.

```text
사용자가 버튼을 눌렀지만, 현재 어느 방에 있는지 확실하지 않다.
방 안이 어두운지 밝은지 정보가 부족하다.
현재 조명이 켜져 있는지 꺼져 있는지 알 수 없다.
사용자가 조명을 켜고 싶은지, 보호자를 부르고 싶은지, 긴급 도움을 요청하려는지 구분하기 어렵다.
```

사용자는 스마트폰 앱에서 메뉴를 고르기 어렵고, 긴 문장으로 설명하기도 어렵다.

이때 시스템이 “아마 조명을 켜려는 것 같다”고 임의로 실행하면 사용자의 의도와 다른 동작이 발생할 수 있다.

반대로 시스템이 단순히 “정보가 부족합니다”라고 말하고 멈춰 버리면, 사용자는 필요한 도움을 받지 못할 수 있다.

따라서 이 시나리오에서는 시스템이 가능한 도움 후보를 짧게 제시하고, 사용자가 선택하거나 보호자가 확인할 수 있도록 돕는다.

이 시나리오의 핵심 상황은 다음과 같다.

```text
사용자의 입력은 발생했지만,
현재 정보만으로 사용자의 의도를 하나로 확정하기 어렵고,
시스템은 가능한 도움 후보를 제시한 뒤,
사용자 또는 보호자의 확인을 통해 적절한 처리 흐름으로 전이해야 한다.
```

## 3. 시스템이 인식해야 하는 상황

시스템은 사용자의 버튼 입력이 있었다는 사실을 확인한다.

하지만 버튼 입력만으로는 사용자의 의도를 확정할 수 없다.

예를 들어 시스템은 다음과 같은 상황을 인식한다.

```text
- 사용자가 버튼을 눌렀다.
- 현재 정보만으로 사용자의 요청을 하나로 특정하기 어렵다.
- 긴급상황 센서가 직접 감지한 위험 정보는 아직 없다.
- 방문자 상황도 아니다.
- 사용자는 긴 문장으로 설명하기 어려울 수 있다.
- 가능한 도움 후보를 짧고 이해하기 쉬운 방식으로 제시해야 한다.
```

이 시나리오에서 LLM은 사용자의 요청을 대신 결정하는 장치가 아니다.

LLM의 역할은 사용자의 입력과 현재 가능한 정보를 바탕으로 **사용자가 선택할 수 있는 후보군을 정리해 제시하는 것**이다.

예를 들어 시스템은 다음과 같은 후보를 제시할 수 있다.

```text
“무엇을 도와드릴까요?
1번은 조명 켜기,
2번은 보호자 호출,
3번은 긴급 도움 요청입니다.”
```

사용자 응답에 따라 흐름은 달라진다.

```text
사용자가 1번을 선택한다.
→ 조명 제어 요청으로 확인
→ Class 1 생활 보조 요청으로 전이

사용자가 2번을 선택한다.
→ 보호자 도움 요청으로 확인
→ 보호자 확인 또는 안전 보류 흐름으로 전이

사용자가 3번을 선택하거나, 추가로 triple-hit 입력을 보낸다.
→ 긴급 도움 요청으로 확인
→ Class 0 긴급상황으로 전이

사용자가 응답하지 못한다.
→ 일정 시간 후 보호자 확인 또는 안전 보류 흐름으로 전이
```

## 4. 정보 출처 노드 및 상태 정보

이 시나리오에서 시스템은 사용자 입력은 받았지만, 요청을 하나로 확정하기에는 일부 상태 정보가 부족하다. 따라서 여러 노드의 정보를 확인하고, LLM이 가능한 후보를 정리하며, 사용자의 선택 또는 보호자 확인을 기다린다.

| 항목 | 정보를 제공하는 노드/구성요소 | 제공되는 정보 | 시나리오 내 상태 |
|---|---|---|---|
| 사용자 입력 | Bounded Input Node | 사용자가 버튼을 눌렀다는 입력 이벤트 | 단일 버튼 입력 발생 |
| 입력 의미 후보 | LLM Guidance Layer 또는 Input Context Mapper | 사용자가 원할 가능성이 있는 도움 후보 | 여러 후보가 존재 |
| 환경 정보 | Context Node | 조도, 점유 여부, 사용자가 있는 공간 등 주변 환경 정보 | 일부 정보가 없거나 불충분함 |
| 기기 상태 | Lighting Actuator Node 또는 Device State Reporter | 조명이 현재 켜져 있는지 꺼져 있는지에 대한 상태 정보 | 상태 확인 불충분 또는 누락 |
| 긴급상황 여부 | Emergency Node | 화재, 가스, 낙상 등 위험 상황 감지 정보 | 직접 감지된 위험 상황 없음 |
| 방문자 여부 | Doorbell/Visitor Context Node | 현관 호출 또는 방문자 감지 정보 | 방문자 상황 아님 |
| 후보 제시 | TTS/Voice Output 또는 Display Output | 사용자가 선택할 수 있는 후보 안내 | 후보 안내 가능 |
| 사용자 선택 | Bounded Input Node, Voice Input 또는 Caregiver Confirmation | 사용자가 후보 중 하나를 선택하거나 보호자가 확인한 결과 | 선택 대기 또는 확인 필요 |
| 요청 재분류 | Mac mini Edge Hub | 사용자 선택 또는 보호자 확인을 바탕으로 한 재판단 | Class 1, Class 0, 또는 안전 보류로 전이 |
| 처리 기록 | Audit Log | 입력, 후보, 선택, 전이 결과 기록 | 후보 제시 및 확인 과정 기록 |

이 표에서 중요한 점은 **LLM이 실행 결정을 내리는 것이 아니라 후보군을 제시한다는 것**이다.

```text
사용자 입력
→ Bounded Input Node가 제공함

후보군 생성
→ LLM Guidance Layer 또는 Input Context Mapper가 지원함

최종 선택
→ 사용자 또는 보호자가 확인함

전이 판단
→ Mac mini Edge Hub가 확인된 선택과 현재 상태를 바탕으로 수행함
```

## 5. 노드별 개발 관점 정리

이 시나리오를 구현하기 위해 필요한 노드와 구성요소는 다음과 같다.

| 노드/구성요소 | 개발 시 필요한 기능 | 이 시나리오에서의 역할 |
|---|---|---|
| Bounded Input Node | 버튼 입력 감지, debounce 처리, 최초 요청 입력과 후보 선택 입력 구분 | 최초 도움 요청과 후보 선택 입력을 전달 |
| Voice Input | 사용자의 짧은 음성 응답 수집 | 사용자가 “1번”, “보호자”, “긴급” 등으로 응답할 수 있는 경우 사용 |
| Context Node | 조도, 점유 여부, 사용자 위치 또는 공간 정보 제공 | 후보를 좁히는 데 필요한 주변 정보 제공 |
| Lighting Actuator Node | 조명 켜기/끄기 수행, 현재 조명 상태 보고 | 후보 중 조명 제어가 선택될 경우 실행 대상 |
| Device State Reporter | 조명 등 기기 상태 보고 | 후보를 좁히거나 실행 가능 여부를 판단하는 데 도움 |
| Emergency Node | 화재, 가스, 낙상 등 위험 상황 감지 | 직접 감지된 위험 상황이 있는 경우 Class 0 전이에 사용 |
| Doorbell/Visitor Context Node | 현관 호출 또는 방문자 감지 여부 제공 | 이 시나리오에서는 방문자 상황이 아님을 나타냄 |
| LLM Guidance Layer | 가능한 도움 후보를 짧고 이해하기 쉬운 문장으로 구성 | 사용자가 선택할 수 있는 후보를 제시 |
| Class 2 Clarification Manager | 후보 제시, 사용자 선택 수집, timeout 처리, 재분류 요청 관리 | Class 2에서 Class 1, Class 0, 안전 보류로 전이하는 흐름을 관리 |
| Mac mini Edge Hub | 입력 수신, 상태 종합, 후보 제시 요청, 선택 확인, 전이 판단 | 정보 부족 상황에서 clarification 흐름을 관리 |
| TTS/Voice Output | 후보군과 확인 요청을 사용자에게 음성으로 안내 | 사용자가 화면을 보지 않아도 후보를 이해할 수 있게 함 |
| Display Output | 후보를 화면 또는 앱에 표시 | 시각적 확인이 가능한 사용자에게 후보 제공 |
| Caregiver Notification 또는 Escalation Interface | 보호자에게 확인 요청 또는 긴급 알림 전달 | 사용자가 직접 선택하기 어렵거나 긴급상황이 확인되는 경우 사용 |
| Audit Log | 입력, 후보, 선택, 전이 결과, 처리 결과 기록 | 후보 제시와 확인 과정을 기록 |

## 6. 처리 과정

### 6.1 사용자가 버튼을 누른다

사용자는 침대나 휠체어에 있는 상태에서 버튼을 한 번 누른다.

Bounded Input Node는 이 입력을 감지한다.

```text
사용자 버튼 입력 1회 발생
```

---

### 6.2 시스템이 상황 정보를 확인한다

Mac mini Edge Hub는 사용자의 입력을 해석하기 위해 주변 상태 정보를 확인한다.

예를 들어 다음과 같은 정보를 확인하려고 한다.

```text
사용자가 어느 공간에 있는가?
방 안이 어두운가?
현재 조명이 켜져 있는가?
최근 사용자가 어떤 요청을 했는가?
긴급상황이나 방문자 상황은 아닌가?
```

하지만 이 시나리오에서는 일부 정보가 충분하지 않다.

---

### 6.3 Class 2 상태로 진입한다

Mac mini Edge Hub는 현재 정보만으로 사용자의 요청을 하나로 확정할 수 없다고 판단한다.

이때 시스템은 Class 2 insufficient-context 상태로 진입한다.

이 상태는 단순 실패가 아니라, 사용자의 의도를 안전하게 확인하기 위한 중간 단계다.

```text
사용자 입력은 있었지만,
현재 정보만으로는 원하는 동작을 하나로 확정하기 어렵다.
→ Class 2 clarification state 진입
```

---

### 6.4 LLM이 가능한 도움 후보를 구성한다

Class 2 Clarification Manager는 LLM Guidance Layer를 통해 사용자가 원할 가능성이 있는 도움 후보를 구성한다.

예를 들어 다음과 같은 후보가 만들어질 수 있다.

```text
1. 조명을 켤까요?
2. 보호자에게 도움을 요청할까요?
3. 긴급상황인가요?
4. 취소하고 대기할까요?
```

후보는 사용자가 쉽게 이해하고 선택할 수 있도록 짧고 명확해야 한다.

---

### 6.5 사용자에게 후보를 안내한다

TTS/Voice Output 또는 Display Output은 후보를 사용자에게 안내한다.

예를 들어 음성 안내는 다음과 같을 수 있다.

```text
“무엇을 도와드릴까요?
1번은 조명 켜기,
2번은 보호자 호출,
3번은 긴급 도움 요청,
4번은 취소입니다.”
```

사용자가 화면을 보기 어렵다면 음성 안내가 중요하고, 사용자가 음성을 듣기 어렵다면 화면, 표시등, 진동 등 다른 방식이 필요할 수 있다.

---

### 6.6 사용자가 후보 중 하나를 선택한다

사용자는 가능한 방식으로 후보 중 하나를 선택한다.

예를 들어 다음과 같은 방법이 가능하다.

```text
버튼을 한 번 더 눌러 1번 선택
버튼을 두 번 눌러 2번 선택
버튼을 세 번 눌러 긴급 도움 요청 선택
짧게 “1번”, “조명”, “보호자”, “긴급”이라고 말하기
보호자가 원격으로 확인하기
```

사용자가 직접 선택하기 어렵거나 일정 시간 응답이 없으면 보호자 확인 또는 안전 보류 흐름으로 넘어갈 수 있다.

---

### 6.7 선택 결과에 따라 Class 1로 전이한다

사용자가 조명 켜기, 조명 끄기와 같은 낮은 위험도의 생활 보조 후보를 선택하면, 시스템은 요청을 Class 1로 전이한다.

예를 들어 사용자가 1번을 선택한 경우 다음과 같이 진행된다.

```text
사용자가 “조명 켜기” 후보를 선택한다.
→ Class 1 생활 보조 요청으로 확인된다.
→ Mac mini Edge Hub가 조명 제어 흐름으로 전이한다.
→ Lighting Actuator Node가 조명 켜기 동작을 수행한다.
```

이 경우 Class 2는 최종 상태가 아니라, Class 1로 전이되기 위한 확인 단계로 동작한다.

---

### 6.8 선택 결과에 따라 Class 0으로 전이한다

사용자가 긴급 도움 요청을 선택하거나, 추가 입력 중 triple-hit 입력이 발생하거나, 센서가 낙상·연기·가스·고온 같은 긴급상황을 감지하면 시스템은 Class 0로 전이할 수 있다.

예를 들어 다음과 같은 상황이다.

```text
사용자가 “긴급 도움 요청”을 선택한다.
사용자가 버튼을 세 번 누른다.
사용자가 “넘어졌어요”라고 말한다.
낙상 감지 노드가 낙상 의심 상태를 감지한다.
연기 또는 가스 감지 노드가 위험 신호를 보낸다.
```

이 경우 시스템은 다음과 같이 진행한다.

```text
긴급상황이 확인된다.
→ Class 0 긴급상황으로 전이한다.
→ 사용자 경고, 보호자 알림, 긴급 대응 흐름으로 진행한다.
```

---

### 6.9 계속 불명확하거나 응답이 없으면 안전 보류 또는 보호자 확인으로 전이한다

사용자가 후보를 선택하지 못하거나, 선택이 계속 불명확하거나, 필요한 정보가 여전히 부족하면 시스템은 임의로 실행하지 않는다.

이때 시스템은 다음 중 하나로 진행할 수 있다.

```text
- 보호자에게 확인 요청을 보낸다.
- 사용자에게 다시 한 번 짧게 안내한다.
- 아무 동작도 실행하지 않고 안전하게 보류한다.
- 일정 시간 후 상태를 다시 확인한다.
```

이 흐름은 사용자의 입력을 무시하는 것이 아니라, 잘못된 실행을 피하면서 도움 연결 가능성을 유지하기 위한 방식이다.

---

### 6.10 처리 결과를 기록한다

Audit Log는 이 과정에서 발생한 주요 정보를 기록한다.

예를 들어 다음과 같은 내용이 기록될 수 있다.

```text
- 버튼 입력이 발생한 시간
- 입력을 제공한 노드
- 부족했던 상황 정보
- Class 2 상태로 진입한 이유
- LLM이 제시한 후보 목록
- 사용자에게 후보가 안내된 방식
- 사용자가 선택한 후보
- Class 1 또는 Class 0로 전이되었는지 여부
- 안전 보류 또는 보호자 확인으로 넘어갔는지 여부
- 최종 처리 결과
```

이 기록은 나중에 시스템이 왜 바로 실행하지 않았는지, 어떤 후보를 제시했는지, 사용자가 어떤 선택을 했는지, 어떤 클래스로 전이되었는지 확인하는 데 사용할 수 있다.

## 7. 예상되는 결과

이 시나리오에서 기대되는 결과는 다음과 같다.

사용자가 버튼을 눌렀지만 요청 의미가 불명확한 경우, 시스템은 바로 실행하지 않고 가능한 후보를 제시한다. 그 후 사용자 또는 보호자의 확인에 따라 Class 1, Class 0, 또는 안전 보류 흐름으로 전이한다.

사용자 입장에서는 다음과 같은 흐름이 된다.

```text
사용자가 버튼을 한 번 누른다.
→ 시스템이 입력을 인식한다.
→ 시스템이 현재 정보만으로 요청을 확정하기 어렵다고 판단한다.
→ Class 2 상태로 진입한다.
→ LLM이 가능한 도움 후보를 구성한다.
→ 시스템이 사용자에게 후보를 안내한다.
→ 사용자가 후보 중 하나를 선택한다.
→ 낮은 위험도의 생활 보조 요청이면 Class 1로 전이한다.
→ 긴급 도움 요청이면 Class 0로 전이한다.
→ 계속 불명확하면 안전 보류 또는 보호자 확인으로 전이한다.
→ 처리 과정이 기록된다.
```

## 8. 사용자에게 주는 의미

이 시나리오는 사용자의 입력을 무시하거나 단순히 실패로 처리하는 상황이 아니다.

오히려 사용자가 복잡하게 설명하기 어려운 상황에서, 시스템이 가능한 선택지를 짧게 정리해 주는 방식이다.

지체장애인이나 고령자는 긴 문장으로 요청을 설명하기 어려울 수 있다. 버튼 하나만 누를 수 있는 상황에서는 시스템이 사용자의 의도를 바로 알기 어려운 경우가 많다.

이때 시스템이 가능한 도움 후보를 제시하면, 사용자는 긴 설명 없이도 원하는 도움을 선택할 수 있다.

또한 사용자가 실제로 긴급한 상황에 있다면, Class 2 확인 과정은 긴급 도움 요청을 발견해 Class 0로 전이하는 통로가 될 수 있다.

이러한 방식은 특히 다음과 같은 상황에서 중요하다.

```text
- 사용자가 버튼만 누를 수 있고 긴 설명을 하기 어려운 상황
- 사용자의 위치나 주변 상태 정보가 부족한 상황
- 같은 버튼 입력이 여러 의미로 해석될 수 있는 상황
- 사용자가 직접 앱 메뉴를 고르기 어려운 상황
- 바로 실행하면 사용자 의도와 다른 동작이 발생할 수 있는 상황
- 처음에는 긴급상황인지 불분명하지만, 확인 과정에서 긴급 도움이 필요하다는 사실이 드러날 수 있는 상황
```

## 9. 개발 관점에서 확인해야 할 점

이 문서는 실제 상황 설명을 중심으로 하지만, 노드 개발을 위해서는 다음 사항을 확인해야 한다.

```text
- Bounded Input Node는 최초 입력과 후보 선택 입력을 구분할 수 있는가?
- Bounded Input Node는 버튼 입력 횟수를 후보 선택 방식으로 사용할 수 있는가?
- Voice Input은 사용자의 짧은 응답을 수집할 수 있는가?
- Context Node는 사용자 위치, 조도, 점유 여부를 충분히 제공할 수 있는가?
- Device State Reporter는 조명과 기기 상태를 최신 상태로 유지할 수 있는가?
- Mac mini Edge Hub는 정보 부족 상황을 감지하고 Class 2로 진입할 수 있는가?
- Class 2 Clarification Manager는 후보 제시, 선택 수집, timeout 처리를 관리할 수 있는가?
- LLM Guidance Layer는 사용자가 이해하기 쉬운 후보를 짧게 제시할 수 있는가?
- TTS/Voice Output은 후보를 순서대로 안내할 수 있는가?
- 사용자는 버튼, 음성, 보호자 확인 등으로 후보를 선택할 수 있는가?
- 선택 결과가 낮은 위험도의 생활 보조이면 Class 1로 전이할 수 있는가?
- 선택 결과가 긴급 도움 요청이거나 추가 긴급 증거가 있으면 Class 0로 전이할 수 있는가?
- 응답이 없거나 계속 불명확하면 안전 보류 또는 보호자 확인으로 전이할 수 있는가?
- Audit Log는 부족한 정보, 제시된 후보, 사용자 선택, 전이 결과, 처리 결과를 기록할 수 있는가?
```

이 체크리스트는 Class 2 clarification / transition 시나리오를 실제 시스템으로 구현할 때 필요한 최소 개발 항목을 정리한 것이다.

## 10. 이 시나리오의 핵심

이 시나리오의 핵심은 다음과 같다.

```text
사용자의 입력은 있었지만,
상황 정보가 부족해 의도를 하나로 확정하기 어려우면,
Class 2 상태에서 LLM이 가능한 도움 후보를 제시하고,
사용자 또는 보호자의 확인을 거쳐
Class 1 생활 보조, Class 0 긴급상황, 또는 안전 보류 흐름으로 전이한다.
```

즉, 이 시나리오는 스마트홈 시스템이 사용자의 제한된 입력을 바로 실패로 처리하지 않고, 가능한 선택지를 제시해 사용자가 원하는 도움을 더 쉽게 선택할 수 있도록 돕는 동시에, 확인 결과에 따라 적절한 처리 클래스로 안전하게 전이하는 기본 Class 2 시나리오다.
