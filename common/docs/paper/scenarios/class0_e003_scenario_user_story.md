# Class 0 E003 Scenario User Story
## 실내 연기 감지로 긴급 대응이 필요한 상황

## 1. 이 문서가 설명하는 JSON 파일

이 문서는 다음 scenario JSON 파일을 실제 사용환경에서 일어날 수 있는 상황으로 풀어 설명한다.

```text
integration/scenarios/class0_e003_scenario_skeleton.json
```

이 JSON 파일은 집 안에서 **연기 감지 상황이 발생했을 때**, 시스템이 이를 긴급상황으로 인식하고 즉시 대응 흐름으로 전환하는지를 검토하기 위한 시나리오다.

여기서는 사용자가 집 안에 있지만 직접 연기 발생 위치를 확인하거나 빠르게 대피하기 어려운 상황에서, Smoke Sensor 또는 Emergency Node가 연기 감지 정보를 전달하는 상황을 예로 든다.

이 문서는 이후 실제 노드를 개발할 때 다음을 확인하는 기준으로도 사용할 수 있다.

```text
- 어떤 노드가 연기 발생을 감지해야 하는가
- 연기 감지 정보는 어떤 방식으로 중앙 시스템에 전달되는가
- 시스템은 일반 생활 요청과 연기 긴급상황을 어떻게 구분하는가
- 사용자가 직접 확인하거나 대피하기 어려운 상황에서 어떤 결과가 기대되는가
```

---

## 2. 발생 상황

사용자는 집 안에 있지만, 현재 직접 빠르게 움직이기 어려운 상태라고 가정한다.

예를 들어 사용자는 침대에 누워 있거나 휠체어에 앉아 있고, 주방이나 거실 쪽에서 발생한 이상 상황을 즉시 확인하기 어렵다.

이때 주방, 거실, 또는 실내 특정 공간의 연기 감지 센서가 연기를 감지한다. 조리 중 음식이 타고 있거나, 전기기기 과열로 연기가 발생하거나, 화재 초기 상황일 수 있다.

사용자는 아직 연기를 직접 보지 못했을 수 있고, 냄새를 감지하지 못했을 수도 있다. 또는 연기를 알아차렸더라도 스마트폰을 조작하거나 음성으로 도움을 요청하기 어려울 수 있다.

따라서 시스템은 연기 감지 정보를 일반적인 생활 보조 요청이 아니라 **긴급상황**으로 판단해야 한다.

이 시나리오는 E001 고온 이상 시나리오와 같은 Class 0 긴급상황이지만, 위험을 포착하는 방식이 다르다. E001은 온도 상승을 통해 과열 또는 화재 진행 가능성을 감지하는 상황이고, E003은 연기 감지를 통해 온도 상승이 심해지기 전의 화재 초기 징후를 포착할 수 있는 상황이다. 따라서 E003은 E001보다 낮은 위급도의 상황이라기보다, **감지 방식과 위험 발생 시점이 다른 긴급상황**으로 이해하는 것이 적절하다.

이 시나리오의 핵심 상황은 다음과 같다.

```text
실내 특정 공간에서 연기가 감지되고,
사용자는 즉시 상황을 확인하거나 대처하기 어려우며,
시스템은 이를 긴급상황으로 인식해야 한다.
```

---

## 3. 시스템이 인식해야 하는 상황

시스템은 사용자의 버튼 입력이나 음성 요청이 없어도, 연기 감지 노드가 제공하는 정보를 확인해야 한다.

예를 들어 시스템은 다음과 같은 정보를 함께 확인한다.

```text
- 실내 특정 공간에서 연기가 감지되었다.
- 연기 감지 정보가 긴급상황 감지 노드에서 전달되었다.
- 사용자가 직접 상황을 확인하거나 빠르게 이동하기 어려울 수 있다.
- 일반적인 조명 제어 요청이나 생활 보조 요청이 아니다.
- 즉각적인 경고와 보호자 알림이 필요할 수 있다.
```

이 정보를 종합하면, 시스템은 이 상황을 **연기 감지에 따른 긴급상황**으로 처리해야 한다.

여기서 중요한 점은 연기 감지 정보가 시스템 안에 저절로 존재하는 것이 아니라는 점이다. 연기 정보는 Smoke Sensor Node 또는 Emergency Node가 실제로 감지하고 전달해야 한다.

Mac mini Edge Hub는 이 긴급상황 정보를 수신한 뒤, 일반적인 사용자 요청 처리 흐름보다 긴급 대응 흐름을 우선해야 한다.

## 4. 정보 출처 노드 및 상태 정보

이 시나리오에서 시스템은 연기 감지 노드가 제공하는 정보를 중심으로 현재 상황을 판단한다. 동시에 사용자 상태, 환경 상태, 안내 출력, 기록 기능도 함께 고려된다.

| 항목 | 정보를 제공하는 노드/구성요소 | 제공되는 정보 | 시나리오 내 상태 |
|---|---|---|---|
| 연기 감지 | Smoke Sensor Node 또는 Emergency Node | 실내 특정 공간의 연기 감지 정보 | 연기 감지됨 |
| 긴급상황 이벤트 | Emergency Node | 연기 감지가 긴급상황에 해당한다는 이벤트 | 긴급 이벤트 발생 |
| 사용자 상태 | Bounded Input Node 또는 사용자 상태 정보 | 사용자가 직접 즉시 대응할 수 있는지와 관련된 입력/상태 | 사용자의 직접 조작 없음 또는 제한적 조작 가능 상태 |
| 환경 정보 | Context Node | 주변 환경 정보, 위치 또는 공간 상태 | 실내 공간에서 이상 상황 발생 |
| 방문자 여부 | Doorbell/Visitor Context Node | 현관 호출 또는 방문자 감지 정보 | 방문자 상황 아님 |
| 요청/상황 해석 | Mac mini Edge Hub | 긴급상황 정보를 종합한 처리 판단 | 연기 감지 긴급상황으로 해석 |
| 사용자 경고 | TTS/Voice Output 또는 Warning Output Node | 사용자에게 위험 상황을 알리는 안내 | 경고 안내 가능 |
| 보호자/외부 알림 | Caregiver Notification 또는 Escalation Interface | 보호자나 관리자에게 상황 전달 | 필요 시 긴급 알림 가능 |
| 처리 기록 | Audit Log | 감지 정보, 처리 판단, 알림 결과 기록 | 긴급상황 처리 결과 기록 |

이 표에서 연기 감지 상태는 단순한 문서상의 상태가 아니라, 실제 센서 노드가 제공하는 위험 신호다.

```text
연기 감지 정보
→ Smoke Sensor Node 또는 Emergency Node가 제공하는 긴급상황 정보

긴급상황 처리 판단
→ Mac mini Edge Hub가 센서 정보를 바탕으로 수행하는 처리 결과
```

따라서 실제 개발에서는 “연기가 감지되었다”는 상태를 단순히 문서에 적는 것이 아니라, **어떤 센서 노드가 어느 공간의 연기를 감지하고, 그 값을 어떤 방식으로 긴급 이벤트로 전달할 것인가**까지 설계해야 한다.

## 5. 노드별 개발 관점 정리

이 시나리오를 구현하기 위해 필요한 노드와 구성요소는 다음과 같다.

| 노드/구성요소 | 개발 시 필요한 기능 | 이 시나리오에서의 역할 |
|---|---|---|
| Smoke Sensor Node | 연기 감지, 센서 상태 전송, 감지 신호 안정화 | 실내 특정 공간의 연기 발생을 감지 |
| Emergency Node | 연기, 고온, 가스, 낙상 등 긴급상황 이벤트 생성 | 연기 감지를 긴급 이벤트로 전달 |
| Context Node | 공간 정보, 주변 환경 정보 제공 | 연기가 발생한 공간과 주변 상태를 이해하는 데 도움 |
| Bounded Input Node | 사용자의 수동 입력 감지 | 이 시나리오에서는 사용자가 직접 입력하지 않아도 됨 |
| Doorbell/Visitor Context Node | 현관 호출 또는 방문자 감지 여부 제공 | 이 시나리오에서는 방문자 상황이 아님을 나타냄 |
| Mac mini Edge Hub | 긴급 이벤트 수신, 상황 판단, 긴급 대응 흐름 실행 | 연기 감지를 Class 0 긴급상황으로 처리 |
| TTS/Voice Output | 사용자에게 위험 상황을 음성으로 안내 | 필요 시 사용자에게 연기 감지 사실을 알림 |
| Warning Output Node | 경고음, 경광등, 화면 알림 등 위험 신호 출력 | 즉각적인 주의 환기 |
| Caregiver Notification 또는 Escalation Interface | 보호자나 관리자에게 알림 전송 | 사용자가 직접 대응하기 어려운 경우 보호자에게 상황 전달 |
| Audit Log | 감지, 판단, 알림, 처리 결과 기록 | 긴급상황 처리 과정 기록 |

## 6. 처리 과정

### 6.1 연기 상태가 감지된다

Smoke Sensor Node 또는 Emergency Node가 실내 특정 공간에서 연기를 감지한다.

이 시나리오에서는 연기가 일반적인 생활 환경에서 기대되는 수준을 넘어 위험 신호로 볼 수 있는 상황을 가정한다.

예를 들어 다음과 같은 의미의 이벤트가 생성된다.

```text
주방 또는 실내 특정 공간에서 연기 감지
```

---

### 6.2 긴급상황 이벤트가 생성된다

Emergency Node는 감지된 연기 정보를 긴급상황 이벤트로 구성한다.

이 이벤트는 일반적인 사용자 요청이 아니라, 즉각적인 확인과 대응이 필요한 위험 신호로 취급된다.

이 시나리오에서 긴급상황 이벤트는 다음 입력 경로로 전달된다.

```text
safe_deferral/emergency/event
```

---

### 6.3 Mac mini Edge Hub가 긴급 이벤트를 수신한다

Mac mini Edge Hub는 Emergency Node가 보낸 연기 감지 긴급 이벤트를 수신한다.

이때 시스템은 사용자가 버튼을 눌렀는지, 음성 명령을 했는지와 무관하게 연기 감지 이벤트를 우선 처리해야 한다.

즉, 이 시나리오는 사용자의 일상 요청을 해석하는 흐름이 아니라, 센서가 감지한 위험 상황을 우선 처리하는 흐름이다.

---

### 6.4 시스템이 긴급상황으로 판단한다

Mac mini Edge Hub는 연기 감지 정보를 바탕으로 현재 상황을 긴급상황으로 판단한다.

이때 시스템은 다음과 같은 점을 함께 고려할 수 있다.

```text
연기가 감지되었다.
일반적인 생활 보조 요청이 아니다.
사용자가 직접 빠르게 확인하거나 대처하기 어려울 수 있다.
즉각적인 경고 또는 보호자 알림이 필요할 수 있다.
```

이 시나리오에서는 이 상황을 연기 감지에 따른 긴급상황으로 해석한다.

---

### 6.5 사용자에게 위험 상황을 알린다

필요한 경우 TTS/Voice Output 또는 Warning Output Node가 사용자에게 위험 상황을 알린다.

예를 들어 다음과 같은 안내가 가능하다.

```text
“주의가 필요합니다. 실내에서 연기가 감지되었습니다.”
```

또는 더 짧은 경고음, 경광등, 화면 알림이 함께 사용될 수 있다.

사용자가 시각·청각·운동 능력에 제한이 있을 수 있으므로, 경고 방식은 여러 형태로 제공될 수 있어야 한다.

---

### 6.6 보호자 또는 관리자에게 알림을 보낼 수 있다

사용자가 직접 상황을 확인하거나 대처하기 어렵다고 판단되는 경우, 시스템은 보호자나 관리자에게 알림을 보낼 수 있다.

예를 들어 다음과 같은 정보가 전달될 수 있다.

```text
- 연기가 감지된 위치
- 감지된 시간
- 연기 감지 상태
- 사용자가 있는 공간 또는 최근 활동 상태
- 시스템이 긴급상황으로 처리했다는 사실
```

이 단계는 사용자가 즉시 대처하기 어려운 상황에서 외부 도움을 빠르게 연결하기 위한 것이다.

---

### 6.7 처리 결과를 기록한다

Audit Log는 이 과정에서 발생한 주요 정보를 기록한다.

예를 들어 다음과 같은 내용이 기록될 수 있다.

```text
- 연기 이벤트가 발생한 시간
- 이벤트를 제공한 노드
- 연기가 감지된 공간
- 시스템이 긴급상황으로 판단한 내용
- 사용자에게 경고가 제공되었는지 여부
- 보호자 또는 관리자에게 알림이 전송되었는지 여부
- 이후 처리 결과
```

이 기록은 나중에 시스템이 정상적으로 긴급상황을 처리했는지 확인하거나, 실험 결과를 분석할 때 사용할 수 있다.

## 7. 예상되는 결과

이 시나리오에서 기대되는 결과는 다음과 같다.

연기 감지가 발생하면 시스템은 이를 일반적인 생활 보조 요청으로 처리하지 않고, 긴급상황으로 인식한다.

사용자 입장에서는 다음과 같은 흐름이 된다.

```text
실내 특정 공간에서 연기가 감지된다.
→ Emergency Node가 긴급 이벤트를 생성한다.
→ Mac mini Edge Hub가 긴급 이벤트를 수신한다.
→ 시스템이 연기 감지 상황을 긴급상황으로 판단한다.
→ 사용자에게 경고가 제공된다.
→ 필요하면 보호자나 관리자에게 알림이 전달된다.
→ 처리 과정이 기록된다.
```

## 8. 사용자에게 주는 의미

이 시나리오는 사용자가 직접 위험을 알아차리거나 즉시 움직이기 어려운 상황에서 중요하다.

지체장애인, 고령자, 또는 움직임이 제한된 사용자는 연기나 화재 초기 상황이 발생했을 때 직접 확인하거나 빠르게 이동하기 어려울 수 있다.

이때 시스템이 먼저 연기 감지를 확인하고, 사용자에게 경고하거나 보호자에게 알림을 보내면 위험 대응 시간을 줄일 수 있다.

이러한 방식은 특히 다음과 같은 사용자에게 도움이 될 수 있다.

```text
- 침대나 휠체어에서 빠르게 이동하기 어려운 사용자
- 주방이나 거실의 이상 상황을 바로 확인하기 어려운 사용자
- 연기나 화재 초기 상황에서 즉시 대피하기 어려운 고령자
- 위험 상황을 음성으로 설명하거나 도움을 요청하기 어려운 사용자
```

## 9. 개발 관점에서 확인해야 할 점

이 문서는 실제 상황 설명을 중심으로 하지만, 노드 개발을 위해서는 다음 사항을 확인해야 한다.

```text
- Smoke Sensor Node는 실내 연기를 안정적으로 감지할 수 있는가?
- Smoke Sensor Node는 오감지와 실제 위험 신호를 구분할 수 있도록 상태 정보를 제공할 수 있는가?
- Emergency Node는 연기 감지 상황을 긴급 이벤트로 변환할 수 있는가?
- Mac mini Edge Hub는 연기 감지 이벤트를 일반 입력보다 우선 처리할 수 있는가?
- TTS/Voice Output은 사용자에게 연기 감지 상황을 즉시 안내할 수 있는가?
- Warning Output Node는 음성 외의 경고 방식을 제공할 수 있는가?
- Caregiver Notification 또는 Escalation Interface는 보호자에게 필요한 정보를 전달할 수 있는가?
- Audit Log는 감지, 판단, 알림, 처리 결과를 빠짐없이 기록할 수 있는가?
```

이 체크리스트는 연기 감지 긴급상황을 실제 시스템으로 구현할 때 필요한 최소 개발 항목을 정리한 것이다.

---

## 10. 이 시나리오의 핵심

이 시나리오의 핵심은 다음과 같다.

```text
사용자가 직접 요청하지 않아도,
긴급상황 감지 노드가 실내 연기를 감지하면,
Mac mini Edge Hub가 이를 우선 처리하고,
사용자와 보호자에게 위험 상황을 알린다.
```

즉, 이 시나리오는 스마트홈 시스템이 사용자의 제한된 조작 능력에만 의존하지 않고, 실내 연기 감지와 같은 위험 상황을 먼저 파악해 안전 대응으로 연결하는 기본 긴급상황 시나리오다.
