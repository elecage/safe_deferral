# Missing-State Fault Scenario User Story
## 필요한 기기 상태 정보가 누락되어 바로 실행하지 않는 상황

## 1. 이 문서가 설명하는 JSON 파일

이 문서는 다음 scenario JSON 파일을 실제 사용환경에서 일어날 수 있는 상황으로 풀어 설명한다.

```text
integration/scenarios/missing_state_scenario_skeleton.json
```

이 JSON 파일은 사용자의 입력과 일부 환경 정보는 들어왔지만, 요청을 안전하게 해석하거나 실행하는 데 필요한 **기기 상태 정보가 누락된 상황**을 다룬다.

여기서는 사용자가 버튼을 눌러 조명 제어 도움을 요청한 것으로 보이지만, 시스템이 현재 조명이 켜져 있는지 꺼져 있는지, 어떤 기기가 정상 응답 중인지 알 수 없어 바로 실행하지 않는 상황을 예로 든다.

이 시나리오는 Class 2 insufficient context와 비슷해 보일 수 있지만, 핵심은 조금 다르다.

```text
Class 2 insufficient context
→ 사용자의 의도나 상황 해석에 필요한 정보가 부족한 상태

Missing-state fault
→ 특정 노드나 기기 상태 보고가 누락되어 실행 가능 여부를 안전하게 판단할 수 없는 상태
```

이 문서는 이후 실제 노드를 개발할 때 다음을 확인하는 기준으로도 사용할 수 있다.

```text
- 어떤 상태 정보가 누락되면 실행을 멈춰야 하는가
- 기기 상태 보고가 없을 때 시스템은 어떻게 안전하게 처리해야 하는가
- 누락된 상태 정보를 사용자에게 어떻게 설명할 것인가
- 상태 보고 장애가 발생했을 때 어떤 보류 또는 보호자 확인 절차가 필요한가
```

---

## 2. 발생 상황

사용자는 침대에 누워 있거나 휠체어에 앉아 있다.

사용자는 집 안의 조명을 켜거나 끄고 싶어 버튼을 한 번 누른다. 버튼 입력 자체는 정상적으로 전달된다.

하지만 현재 시스템은 조명 기기의 상태를 제대로 확인하지 못한다.

예를 들어 다음과 같은 상황일 수 있다.

```text
사용자가 버튼을 눌렀다.
Context Node는 사용자가 실내에 있다는 정보를 제공한다.
방 안의 일부 환경 정보는 전달된다.
하지만 Lighting Actuator Node의 현재 상태 보고가 없다.
Device State Reporter가 최근 상태를 보내지 않았다.
조명이 켜져 있는지 꺼져 있는지 확인할 수 없다.
```

이때 시스템이 상태 확인 없이 조명 제어 명령을 보내면, 사용자의 의도와 다르게 동작하거나 이미 켜진 조명을 다시 켜는 등 불필요한 동작이 발생할 수 있다.

더 중요한 것은, 상태 보고가 누락된 상황에서 시스템이 “정상 상태”라고 가정하면 이후 더 큰 오류를 만들 수 있다는 점이다.

따라서 이 시나리오에서는 시스템이 기기 상태 누락을 감지하고, 바로 실행하지 않고 안전한 보류 또는 확인 흐름으로 넘어가야 한다.

이 시나리오의 핵심 상황은 다음과 같다.

```text
사용자의 입력은 정상적으로 들어왔지만,
실행 대상 기기의 현재 상태가 누락되어 있고,
시스템은 상태를 임의로 가정하지 않고 안전하게 보류해야 한다.
```

---

## 3. 시스템이 인식해야 하는 상황

시스템은 사용자의 버튼 입력이 있었다는 사실을 확인한다.

그러나 실행 대상이 될 수 있는 기기의 현재 상태를 확인하지 못한다.

예를 들어 시스템은 다음과 같은 상황을 인식한다.

```text
- 사용자가 버튼을 눌렀다.
- 일부 환경 정보는 전달되었다.
- 긴급상황은 감지되지 않았다.
- 방문자 상황도 아니다.
- 하지만 조명 또는 기기 상태 정보가 누락되었다.
- 실행 대상 기기가 현재 어떤 상태인지 알 수 없다.
- 따라서 바로 actuator 명령을 보내면 안전하지 않다.
```

이 정보를 종합하면, 시스템은 이 상황을 **missing-state fault에 따른 안전 보류 또는 확인 필요 상황**으로 처리해야 한다.

여기서 중요한 점은 사용자의 입력이 잘못된 것이 아니라는 점이다. 문제는 사용자의 요청이 아니라, 시스템이 그 요청을 처리하기 위해 필요한 상태 정보를 충분히 받지 못했다는 것이다.

Mac mini Edge Hub는 누락된 상태 정보를 임의로 추정하지 않고, 상태 재확인, 사용자 안내, 보호자 확인, 또는 안전 보류 흐름으로 넘어가야 한다.

---

## 4. 정보 출처 노드 및 상태 정보

이 시나리오에서 시스템은 사용자 입력과 일부 환경 정보는 받지만, 실행 판단에 필요한 기기 상태 정보가 누락된다. 따라서 어떤 정보가 제공되었고 어떤 정보가 누락되었는지를 명확히 구분해야 한다.

| 항목 | 정보를 제공하는 노드/구성요소 | 제공되는 정보 | 시나리오 내 상태 |
|---|---|---|---|
| 사용자 입력 | Bounded Input Node | 사용자가 버튼을 눌렀다는 입력 이벤트 | 단일 버튼 입력 발생 |
| 환경 정보 | Context Node | 조도, 점유 여부, 사용자가 있는 공간 등 주변 환경 정보 | 일부 환경 정보 제공됨 |
| 조명 상태 | Lighting Actuator Node 또는 Device State Reporter | 조명이 현재 켜져 있는지 꺼져 있는지에 대한 상태 정보 | 상태 정보 누락 |
| 기기 응답 상태 | Lighting Actuator Node 또는 Device Health Reporter | 실행 대상 기기가 응답 가능한지 여부 | 응답 상태 불확실 |
| 긴급상황 여부 | Emergency Node | 화재, 가스, 낙상 등 위험 상황 감지 정보 | 위험 상황 없음 |
| 방문자 여부 | Doorbell/Visitor Context Node | 현관 호출 또는 방문자 감지 정보 | 방문자 상황 아님 |
| 누락 상태 판단 | Mac mini Edge Hub | 필요한 상태 정보가 누락되었다는 판단 | missing-state fault로 판단 |
| 사용자 안내 | TTS/Voice Output 또는 Display Output | 상태 확인이 필요함을 사용자에게 안내 | 안내 가능 |
| 상태 재확인 | Device State Reporter 또는 Health Check Routine | 누락된 상태 정보를 다시 요청하거나 확인 | 재확인 필요 |
| 보호자/관리자 확인 | Caregiver Notification 또는 Escalation Interface | 상태 누락으로 자동 실행하지 않았음을 전달 | 필요 시 확인 요청 |
| 처리 기록 | Audit Log | 입력, 누락 상태, 보류 또는 확인 결과 기록 | missing-state 처리 결과 기록 |

이 표에서 핵심은 **입력 자체가 없는 것이 아니라, 실행에 필요한 상태 정보가 없는 것**이다.

```text
사용자 입력
→ Bounded Input Node가 제공함

일부 환경 정보
→ Context Node가 제공함

누락된 기기 상태
→ Lighting Actuator Node 또는 Device State Reporter가 제공하지 못함

missing-state 판단
→ Mac mini Edge Hub가 필요한 상태 정보의 부재를 감지함
```

따라서 실제 개발에서는 단순히 “입력이 들어왔는가”뿐 아니라, **그 입력을 실행하기 전에 필요한 상태 정보가 모두 준비되었는가**를 확인해야 한다.

## 5. 노드별 개발 관점 정리

이 시나리오를 구현하기 위해 필요한 노드와 구성요소는 다음과 같다.

| 노드/구성요소 | 개발 시 필요한 기능 | 이 시나리오에서의 역할 |
|---|---|---|
| Bounded Input Node | 버튼 입력 감지, debounce 처리, 입력 이벤트 생성 | 사용자가 버튼을 눌렀다는 사실을 시스템에 전달 |
| Context Node | 조도, 점유 여부, 사용자 위치 또는 공간 정보 제공 | 일부 환경 정보를 제공하지만, 기기 상태 누락을 대체하지는 못함 |
| Lighting Actuator Node | 조명 켜기/끄기 수행, 현재 조명 상태 보고, 실행 결과 ACK 반환 | 이 시나리오에서는 상태 보고가 누락되거나 확인되지 않음 |
| Device State Reporter | 조명 등 기기 상태를 주기적 또는 이벤트 기반으로 보고 | 상태 누락 여부가 이 시나리오의 핵심 조건 |
| Device Health Reporter | 기기 온라인 상태, 마지막 응답 시간, 통신 상태 보고 | 실행 대상 기기가 정상 응답 가능한지 확인 |
| Emergency Node | 화재, 가스, 낙상 등 위험 상황 감지 | 이 시나리오에서는 위험 상황이 없음을 나타냄 |
| Doorbell/Visitor Context Node | 현관 호출 또는 방문자 감지 여부 제공 | 이 시나리오에서는 방문자 상황이 아님을 나타냄 |
| Mac mini Edge Hub | 입력 수신, 상태 누락 판단, 실행 보류, 재확인 요청, 로그 기록 | 상태 누락 시 임의 실행을 막고 안전한 보류로 전환 |
| TTS/Voice Output | 사용자에게 상태 확인 중임을 안내 | 필요 시 “기기 상태를 확인하고 있습니다.” 안내 |
| Display Output | 상태 확인 또는 실행 보류 메시지 표시 | 시각적 확인이 가능한 사용자에게 안내 |
| Caregiver Notification 또는 Escalation Interface | 보호자나 관리자에게 상태 누락 또는 확인 필요 상황 전달 | 사용자가 직접 대응하기 어려운 경우 도움 연결 |
| Audit Log | 입력, 누락된 상태, 보류 이유, 재확인 결과 기록 | 왜 실행하지 않았는지 추적 가능하게 함 |

## 6. 처리 과정

### 6.1 사용자가 버튼을 누른다

사용자는 침대나 휠체어에 있는 상태에서 버튼을 한 번 누른다.

Bounded Input Node는 이 입력을 감지한다.

```text
사용자 버튼 입력 1회 발생
```

---

### 6.2 시스템이 환경 정보를 확인한다

Mac mini Edge Hub는 입력을 해석하기 위해 주변 상태 정보를 확인한다.

예를 들어 Context Node는 사용자가 실내에 있고, 방 안이 어둡다는 일부 정보를 제공할 수 있다.

```text
사용자 입력 확인
일부 환경 정보 확인
```

---

### 6.3 실행 대상 기기 상태를 확인하려고 한다

시스템은 조명 제어 후보가 가능한지 확인하기 위해 Lighting Actuator Node 또는 Device State Reporter의 상태 정보를 확인하려고 한다.

필요한 정보는 다음과 같다.

```text
현재 조명이 켜져 있는가?
현재 조명이 꺼져 있는가?
조명 노드는 온라인인가?
마지막 상태 보고는 언제 들어왔는가?
명령을 보내면 응답할 수 있는가?
```

---

### 6.4 필요한 상태 정보가 누락되었음을 확인한다

이 시나리오에서는 실행 대상 기기의 상태 정보가 누락되어 있다.

예를 들어 다음과 같은 상황이다.

```text
조명 상태 정보가 없다.
Device State Reporter의 최근 보고가 없다.
Lighting Actuator Node의 응답 상태가 불확실하다.
```

따라서 시스템은 현재 상태를 알 수 없는 기기에 대해 바로 명령을 보내지 않는다.

---

### 6.5 시스템이 missing-state fault로 판단한다

Mac mini Edge Hub는 필요한 상태 정보가 누락되었다고 판단한다.

이 단계에서 시스템은 다음과 같이 처리한다.

```text
사용자 입력은 정상적으로 들어왔다.
하지만 실행 대상 기기 상태가 확인되지 않는다.
상태를 임의로 가정하면 잘못된 실행이 발생할 수 있다.
→ missing-state fault로 판단하고 자동 실행을 보류한다.
```

---

### 6.6 사용자에게 상태 확인 중임을 안내한다

필요한 경우 TTS/Voice Output 또는 Display Output이 사용자에게 현재 상태를 안내한다.

예를 들어 다음과 같은 안내가 가능하다.

```text
“기기 상태를 확인하고 있습니다.”
```

또는 다음과 같이 더 명확히 안내할 수 있다.

```text
“현재 조명 상태를 확인할 수 없어 바로 실행하지 않았습니다.”
```

이 안내는 사용자가 자신의 요청이 무시된 것이 아니라, 시스템이 안전을 위해 확인 중이라는 것을 알 수 있게 해준다.

---

### 6.7 상태 재확인을 시도한다

시스템은 Device State Reporter 또는 Health Check Routine을 통해 누락된 상태 정보를 다시 요청할 수 있다.

예를 들어 다음과 같은 재확인을 수행할 수 있다.

```text
조명 상태 재요청
기기 온라인 여부 확인
마지막 상태 보고 시간 확인
기기 응답 가능 여부 확인
```

상태가 회복되면 이후 사용자 확인 또는 안전한 실행 흐름으로 이어질 수 있다.

---

### 6.8 상태가 계속 누락되면 안전 보류 또는 보호자 확인으로 넘어간다

상태 정보가 계속 들어오지 않거나 기기 응답이 불확실하면 시스템은 자동 실행하지 않는다.

이때 시스템은 다음 중 하나로 진행할 수 있다.

```text
- 보호자 또는 관리자에게 확인 요청을 보낸다.
- 사용자에게 다시 안내한다.
- 아무 동작도 실행하지 않고 안전하게 보류한다.
- 일정 시간 후 상태를 다시 확인한다.
```

이 흐름은 사용자의 요청을 무시하는 것이 아니라, 상태가 확인되지 않은 기기를 임의로 제어하지 않기 위한 안전한 처리 방식이다.

---

### 6.9 처리 결과를 기록한다

Audit Log는 이 과정에서 발생한 주요 정보를 기록한다.

예를 들어 다음과 같은 내용이 기록될 수 있다.

```text
- 버튼 입력이 발생한 시간
- 입력을 제공한 노드
- 확인하려고 한 기기 상태
- 누락된 상태 정보
- 마지막 상태 보고 시간
- 시스템이 자동 실행하지 않은 이유
- 재확인 시도 여부
- 사용자 안내 여부
- 보호자 또는 관리자 확인 요청 여부
- 최종 보류 또는 회복 결과
```

이 기록은 나중에 시스템이 왜 자동 실행하지 않았는지, 어떤 상태 정보가 누락되었는지, 어떤 노드를 개선해야 하는지 확인하는 데 사용할 수 있다.

---

## 7. 예상되는 결과

이 시나리오에서 기대되는 결과는 다음과 같다.

사용자의 입력은 정상적으로 들어왔지만, 실행 대상 기기의 상태 정보가 누락되면 시스템은 해당 동작을 바로 실행하지 않는다.

사용자 입장에서는 다음과 같은 흐름이 된다.

```text
사용자가 버튼을 한 번 누른다.
→ 시스템이 입력을 인식한다.
→ 시스템이 환경 정보를 일부 확인한다.
→ 시스템이 실행 대상 기기의 상태를 확인하려고 한다.
→ 필요한 상태 정보가 누락되었음을 확인한다.
→ 시스템이 자동 실행하지 않고 상태 확인 또는 보류로 전환한다.
→ 필요하면 사용자에게 상태 확인 중임을 안내한다.
→ 상태가 회복되면 후속 처리로 이어질 수 있다.
→ 상태가 계속 누락되면 보호자 확인 또는 안전 보류로 이어진다.
→ 처리 과정이 기록된다.
```

---

## 8. 사용자에게 주는 의미

이 시나리오는 시스템이 사용자의 입력을 무시하는 상황이 아니다.

오히려 사용자의 요청을 잘못 처리하지 않기 위해, 필요한 기기 상태를 확인하는 안전 절차다.

지체장애인이나 고령자는 스마트홈 시스템에 의존해 조명, 알림, 도움 요청 등을 수행할 수 있다. 이때 시스템이 기기의 상태를 모른 채 자동으로 실행하면, 예상하지 못한 동작이나 실패가 발생할 수 있다.

따라서 missing-state fault 시나리오는 시스템이 다음 원칙을 지키도록 한다.

```text
상태를 모르면 임의로 실행하지 않는다.
상태를 다시 확인한다.
사용자나 보호자에게 필요한 경우 안내한다.
왜 실행하지 않았는지 기록한다.
```

이러한 방식은 특히 다음과 같은 상황에서 중요하다.

```text
- 조명 상태 보고가 끊긴 상황
- 기기 온라인 여부가 불확실한 상황
- 센서 또는 actuator 노드가 일시적으로 응답하지 않는 상황
- 마지막 상태 보고가 오래된 상황
- 사용자가 직접 기기 상태를 확인하기 어려운 상황
```

## 9. 개발 관점에서 확인해야 할 점

이 문서는 실제 상황 설명을 중심으로 하지만, 노드 개발을 위해서는 다음 사항을 확인해야 한다.

```text
- Bounded Input Node는 버튼 입력을 안정적으로 감지할 수 있는가?
- Context Node는 일부 환경 정보를 제공하되, 기기 상태 누락을 명확히 구분할 수 있는가?
- Lighting Actuator Node는 현재 조명 상태를 주기적 또는 이벤트 기반으로 보고할 수 있는가?
- Device State Reporter는 마지막 상태 보고 시간과 최신 상태를 제공할 수 있는가?
- Device Health Reporter는 기기 온라인 여부와 응답 가능성을 제공할 수 있는가?
- Mac mini Edge Hub는 필요한 상태 정보가 누락되었는지 감지할 수 있는가?
- Mac mini Edge Hub는 상태 누락 시 자동 실행을 막을 수 있는가?
- 상태 재확인 루틴은 누락된 상태를 다시 요청할 수 있는가?
- TTS/Voice Output은 사용자에게 상태 확인 중임을 안내할 수 있는가?
- Caregiver Notification 또는 Escalation Interface는 상태 누락이 계속될 때 보호자 확인으로 연결할 수 있는가?
- Audit Log는 누락된 상태 정보, 마지막 보고 시간, 보류 이유, 재확인 결과를 기록할 수 있는가?
```

이 체크리스트는 missing-state fault 상황을 실제 시스템으로 구현할 때 필요한 최소 개발 항목을 정리한 것이다.

## 10. 이 시나리오의 핵심

이 시나리오의 핵심은 다음과 같다.

```text
사용자의 입력은 정상적으로 들어왔지만,
실행 대상 기기의 상태 정보가 누락되었으면,
시스템은 상태를 임의로 가정하지 않고,
자동 실행을 보류한 뒤 상태 재확인, 사용자 안내, 보호자 확인 또는 안전 보류로 전환한다.
```

즉, 이 시나리오는 스마트홈 시스템이 사용자의 제한된 입력을 처리할 때, 필요한 상태 정보가 누락된 상태에서 성급하게 실행하지 않고 안전하게 상태를 확인하도록 하는 missing-state fault 시나리오다.
