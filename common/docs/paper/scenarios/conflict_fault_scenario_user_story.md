# Conflict Fault Scenario User Story
## 여러 생활 보조 후보가 동시에 가능해 바로 실행하지 않는 상황

## 1. 이 문서가 설명하는 JSON 파일

이 문서는 다음 scenario JSON 파일을 실제 사용환경에서 일어날 수 있는 상황으로 풀어 설명한다.

```text
integration/scenarios/conflict_fault_scenario_skeleton.json
```

이 JSON 파일은 사용자의 입력과 주변 정보가 들어왔지만, 시스템이 해석할 수 있는 **생활 보조 후보가 여러 개 동시에 존재하는 상황**을 다룬다.

여기서는 사용자가 버튼을 눌렀을 때, 시스템이 “거실 조명을 켜려는 것인지”, “침실 조명을 켜려는 것인지”, 또는 “다른 낮은 위험도의 도움을 원하는 것인지”를 하나로 확정하기 어려운 상황을 예로 든다.

이 시나리오는 일반적인 정보 부족 상황과 비슷해 보일 수 있지만, 핵심은 조금 다르다.

```text
Class 2 insufficient context
→ 필요한 정보가 부족해서 후보를 확정하기 어려운 상황

Conflict fault
→ 정보는 일부 존재하지만, 여러 후보가 동시에 그럴듯해서 하나를 고르면 잘못 실행될 수 있는 상황
```

이 문서는 이후 실제 노드를 개발할 때 다음을 확인하는 기준으로도 사용할 수 있다.

```text
- 어떤 상황에서 여러 생활 보조 후보가 동시에 발생할 수 있는가
- 시스템은 후보 충돌을 어떻게 감지해야 하는가
- 후보가 충돌할 때 바로 실행하지 않고 어떻게 안전하게 처리해야 하는가
- 사용자가 원하는 후보를 확인하기 위해 어떤 인터랙션이 필요한가
```

---

## 2. 발생 상황

사용자는 침대에 누워 있거나 휠체어에 앉아 있다.

사용자는 버튼을 한 번 눌러 집 안의 조명 제어 도움을 요청하려고 한다.

하지만 현재 시스템에는 다음과 같은 애매한 상황이 존재한다.

```text
사용자가 있는 위치가 침실인지 거실인지 확실하지 않다.
침실 조명과 거실 조명이 모두 꺼져 있다.
최근 사용자가 거실 조명을 켠 적도 있고, 침실 조명을 켠 적도 있다.
버튼 입력 하나만으로는 어떤 조명을 제어하려는지 확정하기 어렵다.
```

즉, 시스템은 사용자의 입력을 조명 제어 요청으로 볼 수는 있지만, **어떤 조명을 제어해야 하는지**가 충돌한다.

이때 시스템이 임의로 거실 조명을 켜거나 침실 조명을 켜면, 사용자의 실제 의도와 다른 동작이 발생할 수 있다.

따라서 이 시나리오에서는 시스템이 여러 후보 중 하나를 임의로 선택하지 않고, 안전하게 보류하거나 사용자 확인을 요청해야 한다.

이 시나리오의 핵심 상황은 다음과 같다.

```text
사용자의 입력은 생활 보조 요청으로 보이지만,
동시에 여러 실행 후보가 가능하고,
시스템은 임의 선택 대신 안전한 확인 또는 보류 흐름으로 처리해야 한다.
```

---

## 3. 시스템이 인식해야 하는 상황

시스템은 사용자의 버튼 입력이 있었다는 사실을 확인한다.

또한 현재 상황을 보면 조명 제어 요청일 가능성도 있다.

하지만 후보가 하나로 좁혀지지 않는다.

예를 들어 시스템은 다음과 같은 상황을 인식한다.

```text
- 사용자가 버튼을 눌렀다.
- 조명 제어 요청일 가능성이 있다.
- 침실 조명과 거실 조명 모두 후보가 될 수 있다.
- 두 후보 모두 낮은 위험도의 생활 보조 요청으로 볼 수 있다.
- 하지만 어느 쪽이 사용자의 실제 의도인지 확정하기 어렵다.
- 긴급상황은 감지되지 않았다.
- 방문자 상황도 아니다.
```

이 정보를 종합하면, 시스템은 이 상황을 **후보 충돌에 따른 안전 보류 또는 확인 필요 상황**으로 처리해야 한다.

여기서 중요한 점은 후보들이 모두 위험한 것은 아니라는 점이다. 조명 켜기 자체는 낮은 위험도의 생활 보조일 수 있다. 하지만 여러 후보 중 잘못된 것을 선택하면 사용자의 의도와 다른 결과가 발생한다.

따라서 시스템은 “가능한 후보가 있다”는 이유만으로 자동 실행하지 않고, 충돌을 감지해 확인 흐름으로 넘어가야 한다.

## 4. 정보 출처 노드 및 상태 정보

이 시나리오에서 시스템은 사용자 입력과 여러 상태 정보를 함께 확인한다. 문제는 정보가 완전히 없는 것이 아니라, 여러 후보를 동시에 지지하는 정보가 존재한다는 점이다.

| 항목 | 정보를 제공하는 노드/구성요소 | 제공되는 정보 | 시나리오 내 상태 |
|---|---|---|---|
| 사용자 입력 | Bounded Input Node | 사용자가 버튼을 눌렀다는 입력 이벤트 | 단일 버튼 입력 발생 |
| 입력 의미 후보 | Input Context Mapper 또는 LLM Guidance Layer | 사용자가 원할 가능성이 있는 후보 목록 | 여러 후보가 동시에 존재 |
| 사용자 위치 | Context Node 또는 Occupancy/Location Node | 사용자가 어느 공간에 있는지에 대한 정보 | 위치 정보가 모호하거나 상충됨 |
| 조명 상태 | Lighting Actuator Node 또는 Device State Reporter | 침실 조명과 거실 조명의 현재 상태 | 여러 조명이 제어 후보가 될 수 있음 |
| 최근 사용 맥락 | Context History 또는 Audit Log | 최근 사용 패턴, 최근 조명 제어 이력 | 여러 후보를 동시에 지지할 수 있음 |
| 긴급상황 여부 | Emergency Node | 화재, 가스, 낙상 등 위험 상황 감지 정보 | 위험 상황 없음 |
| 방문자 여부 | Doorbell/Visitor Context Node | 현관 호출 또는 방문자 감지 정보 | 방문자 상황 아님 |
| 후보 충돌 판단 | Mac mini Edge Hub | 여러 후보가 동시에 가능하다는 판단 | conflict fault로 판단 |
| 사용자 확인 | TTS/Voice Output 또는 Display Output | 어떤 후보를 원하는지 묻는 안내 | 확인 안내 가능 |
| 사용자 선택 | Bounded Input Node, Voice Input 또는 Caregiver Confirmation | 사용자가 선택한 후보 또는 보호자 확인 결과 | 선택 대기 또는 확인 필요 |
| 처리 기록 | Audit Log | 입력, 후보 충돌, 확인 요청, 처리 결과 기록 | 충돌 처리 결과 기록 |

이 표에서 핵심은 **후보가 없어서 멈추는 것이 아니라, 후보가 여러 개라서 멈춘다는 점**이다.

```text
사용자 입력
→ Bounded Input Node가 제공함

여러 후보
→ Context Node, Device State Reporter, Context History 등이 동시에 여러 가능성을 만듦

후보 충돌 판단
→ Mac mini Edge Hub가 후보를 비교하고 충돌 상태로 판단함

최종 선택
→ 사용자 또는 보호자 확인을 통해 결정함
```

## 5. 노드별 개발 관점 정리

이 시나리오를 구현하기 위해 필요한 노드와 구성요소는 다음과 같다.

| 노드/구성요소 | 개발 시 필요한 기능 | 이 시나리오에서의 역할 |
|---|---|---|
| Bounded Input Node | 버튼 입력 감지, debounce 처리, 후보 선택 입력 감지 | 최초 입력과 이후 후보 선택 입력을 전달 |
| Input Context Mapper | 입력과 현재 상태를 연결해 가능한 후보 목록 생성 | 여러 조명 제어 후보를 구성 |
| LLM Guidance Layer | 후보를 사용자에게 이해하기 쉬운 문장으로 정리 | 충돌 후보를 안내 문장으로 변환 |
| Context Node | 조도, 점유 여부, 사용자 위치 또는 공간 정보 제공 | 사용자의 위치와 주변 상황을 제공 |
| Occupancy/Location Node | 사용자가 어느 방에 있는지 추정 | 후보를 좁히는 데 필요하지만, 이 시나리오에서는 모호할 수 있음 |
| Lighting Actuator Node | 조명 켜기/끄기 수행, 현재 조명 상태 보고 | 여러 조명이 실행 후보로 등장 |
| Device State Reporter | 조명 등 기기 상태를 보고 | 각 후보의 현재 상태를 제공 |
| Context History 또는 Audit Log | 최근 동작 이력, 사용 패턴 기록 | 최근 맥락이 여러 후보를 동시에 지지할 수 있음 |
| Emergency Node | 위험 상황 감지 | 이 시나리오에서는 위험 상황이 없음을 나타냄 |
| Doorbell/Visitor Context Node | 방문자 상황 여부 제공 | 이 시나리오에서는 방문자 상황이 아님을 나타냄 |
| Mac mini Edge Hub | 후보 충돌 감지, 확인 흐름 관리, 최종 처리 판단 | 여러 후보 중 임의 실행을 막고 안전한 확인으로 전환 |
| TTS/Voice Output | 사용자에게 후보 확인 질문을 안내 | “거실 조명을 켤까요, 침실 조명을 켤까요?” 안내 |
| Display Output | 후보를 화면 또는 앱에 표시 | 시각적 선택이 가능한 사용자에게 후보 제공 |
| Caregiver Notification 또는 Escalation Interface | 사용자가 직접 선택하지 못하는 경우 보호자 확인 요청 | 충돌이 해결되지 않을 때 도움 연결 |
| Audit Log | 후보 목록, 충돌 판단, 사용자 선택, 처리 결과 기록 | 왜 자동 실행하지 않았는지 기록 |

## 6. 처리 과정

### 6.1 사용자가 버튼을 누른다

사용자는 침대나 휠체어에 있는 상태에서 버튼을 한 번 누른다.

Bounded Input Node는 이 입력을 감지한다.

```text
사용자 버튼 입력 1회 발생
```

---

### 6.2 시스템이 가능한 요청 후보를 만든다

Mac mini Edge Hub는 입력과 주변 상태를 확인한다.

Input Context Mapper 또는 LLM Guidance Layer는 현재 상황에서 가능한 후보를 만든다.

예를 들어 다음과 같은 후보가 생길 수 있다.

```text
1. 침실 조명을 켠다.
2. 거실 조명을 켠다.
3. 아무 동작도 하지 않고 대기한다.
```

---

### 6.3 여러 후보가 동시에 가능하다는 것을 확인한다

시스템은 후보를 비교한다.

이 시나리오에서는 침실 조명과 거실 조명이 모두 그럴듯한 후보로 남아 있다.

```text
침실 조명 켜기 후보가 가능하다.
거실 조명 켜기 후보도 가능하다.
두 후보 중 하나를 확정할 정보가 부족하거나 상충된다.
```

이 상황은 단순한 정보 부족과 다르다. 시스템은 가능한 후보를 찾았지만, 여러 후보가 동시에 가능해 하나를 고를 수 없는 상태다.

---

### 6.4 시스템이 임의 실행을 하지 않는다

Mac mini Edge Hub는 후보 중 하나를 임의로 선택하지 않는다.

예를 들어 침실 조명을 켤 수도 있고 거실 조명을 켤 수도 있지만, 사용자의 실제 의도가 확인되지 않았기 때문에 자동 실행하지 않는다.

이 단계에서 시스템은 다음과 같이 판단한다.

```text
여러 후보가 동시에 가능하다.
임의로 하나를 실행하면 사용자 의도와 다를 수 있다.
→ 안전한 확인 또는 보류 흐름으로 전환한다.
```

---

### 6.5 사용자에게 후보를 확인한다

TTS/Voice Output 또는 Display Output이 사용자에게 후보를 안내한다.

예를 들어 다음과 같은 안내가 가능하다.

```text
“어떤 조명을 켤까요?
1번은 침실 조명,
2번은 거실 조명,
3번은 취소입니다.”
```

사용자는 버튼 입력, 짧은 음성 응답, 화면 선택 등 가능한 방식으로 후보 중 하나를 선택할 수 있다.

---

### 6.6 사용자가 후보를 선택한다

사용자는 가능한 방식으로 원하는 후보를 선택한다.

예를 들어 다음과 같은 방법이 가능하다.

```text
버튼을 한 번 더 눌러 1번 선택
버튼을 두 번 눌러 2번 선택
짧게 “침실” 또는 “거실”이라고 말하기
보호자가 원격으로 확인하기
```

---

### 6.7 선택된 후보가 안전하면 실행한다

사용자가 후보 중 하나를 명확히 선택하면, Mac mini Edge Hub는 선택된 후보가 안전하게 실행 가능한지 확인한다.

예를 들어 사용자가 “침실 조명”을 선택했다면 Lighting Actuator Node에 침실 조명 켜기 명령을 보낸다.

```text
사용자가 침실 조명을 선택한다.
→ 침실 조명 제어 요청으로 확정된다.
→ Lighting Actuator Node가 침실 조명을 켠다.
```

---

### 6.8 계속 충돌하거나 응답이 없으면 안전 보류한다

사용자가 응답하지 않거나, 선택이 여전히 불분명하거나, 보호자 확인도 되지 않으면 시스템은 자동 실행하지 않는다.

이때 시스템은 다음 중 하나로 진행할 수 있다.

```text
- 다시 짧게 확인 질문을 제공한다.
- 보호자에게 확인 요청을 보낸다.
- 아무 동작도 실행하지 않고 안전하게 보류한다.
- 일정 시간 후 상태를 다시 확인한다.
```

이 흐름은 사용자의 요청을 무시하는 것이 아니라, 잘못된 후보 실행을 막기 위한 안전한 처리 방식이다.

---

### 6.9 처리 결과를 기록한다

Audit Log는 이 과정에서 발생한 주요 정보를 기록한다.

예를 들어 다음과 같은 내용이 기록될 수 있다.

```text
- 버튼 입력이 발생한 시간
- 입력을 제공한 노드
- 생성된 후보 목록
- 충돌한 후보들
- 시스템이 임의 실행하지 않은 이유
- 사용자에게 제시한 확인 질문
- 사용자가 선택한 후보
- 실행 또는 보류 결과
```

이 기록은 나중에 시스템이 왜 자동 실행하지 않았는지, 어떤 후보들이 충돌했는지, 최종적으로 어떤 처리가 되었는지 확인하는 데 사용할 수 있다.

---

## 7. 예상되는 결과

이 시나리오에서 기대되는 결과는 다음과 같다.

사용자의 입력에 대해 여러 후보가 동시에 가능하면, 시스템은 하나를 임의로 실행하지 않는다. 대신 사용자에게 후보를 안내하고, 사용자가 선택하거나 보호자가 확인한 뒤에만 실행한다.

사용자 입장에서는 다음과 같은 흐름이 된다.

```text
사용자가 버튼을 한 번 누른다.
→ 시스템이 입력을 인식한다.
→ 시스템이 가능한 도움 후보를 만든다.
→ 침실 조명과 거실 조명 등 여러 후보가 동시에 가능하다고 판단한다.
→ 시스템이 자동 실행하지 않고 후보를 사용자에게 안내한다.
→ 사용자가 원하는 후보를 선택한다.
→ 선택된 후보가 안전하면 해당 조명이 제어된다.
→ 선택이 없거나 계속 불명확하면 안전하게 보류된다.
→ 처리 과정이 기록된다.
```

---

## 8. 사용자에게 주는 의미

이 시나리오는 시스템이 사용자의 입력을 무시하는 것이 아니다.

오히려 여러 해석이 가능한 상황에서 사용자의 실제 의도를 확인하려는 과정이다.

지체장애인이나 고령자는 버튼 하나만으로 여러 스마트홈 기능을 사용해야 할 수 있다. 하지만 버튼 입력 하나가 항상 같은 의미를 가지는 것은 아니다. 사용자가 어디에 있는지, 어떤 조명이 켜져 있는지, 최근 어떤 행동을 했는지에 따라 입력의 의미가 달라질 수 있다.

이때 시스템이 후보 중 하나를 임의로 실행하면 사용자가 원하지 않는 동작이 발생할 수 있다.

따라서 conflict fault 시나리오는 시스템이 **가능한 후보가 여러 개일 때도 신중하게 확인하고 처리해야 한다**는 점을 보여준다.

이러한 방식은 특히 다음과 같은 상황에서 중요하다.

```text
- 같은 버튼 입력이 여러 조명 제어 후보로 해석될 수 있는 상황
- 사용자의 위치 정보가 모호한 상황
- 여러 기기 상태가 동시에 제어 후보가 되는 상황
- 사용자가 긴 문장으로 정확한 대상을 말하기 어려운 상황
- 잘못된 자동 실행이 사용자에게 불편을 줄 수 있는 상황
```

## 9. 개발 관점에서 확인해야 할 점

이 문서는 실제 상황 설명을 중심으로 하지만, 노드 개발을 위해서는 다음 사항을 확인해야 한다.

```text
- Bounded Input Node는 최초 입력과 후보 선택 입력을 구분할 수 있는가?
- Input Context Mapper는 입력에서 가능한 후보 목록을 생성할 수 있는가?
- Context Node는 사용자 위치와 공간 정보를 제공할 수 있는가?
- Occupancy/Location Node는 사용자가 어느 공간에 있는지 추정할 수 있는가?
- Lighting Actuator Node는 여러 조명의 상태를 보고할 수 있는가?
- Device State Reporter는 각 조명의 최신 상태를 유지할 수 있는가?
- Mac mini Edge Hub는 여러 후보가 동시에 가능한 conflict 상태를 감지할 수 있는가?
- LLM Guidance Layer는 충돌 후보를 사용자가 이해하기 쉬운 문장으로 제시할 수 있는가?
- TTS/Voice Output은 후보를 짧고 명확하게 안내할 수 있는가?
- 사용자는 버튼, 음성, 보호자 확인 등으로 후보를 선택할 수 있는가?
- 응답이 없거나 계속 충돌하면 시스템은 안전하게 보류할 수 있는가?
- Audit Log는 후보 충돌, 사용자 선택, 보류 또는 실행 결과를 기록할 수 있는가?
```

이 체크리스트는 conflict fault 상황을 실제 시스템으로 구현할 때 필요한 최소 개발 항목을 정리한 것이다.

## 10. 이 시나리오의 핵심

이 시나리오의 핵심은 다음과 같다.

```text
사용자의 입력에 대해 여러 생활 보조 후보가 동시에 가능하면,
시스템은 임의로 하나를 실행하지 않고,
사용자 또는 보호자의 확인을 통해 후보를 선택한 뒤,
확인된 후보만 안전하게 처리한다.
```

즉, 이 시나리오는 스마트홈 시스템이 사용자의 제한된 입력을 해석할 때, 여러 후보가 동시에 가능한 상황에서도 성급하게 자동 실행하지 않고 안전한 확인 절차를 거치도록 하는 conflict fault 시나리오다.
